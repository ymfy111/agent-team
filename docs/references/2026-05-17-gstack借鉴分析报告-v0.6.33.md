# gstack 借鉴分析报告

> 版本：v0.6.33  
> 日期：2026-05-17  
> 状态：外部参考分析 / 已转化为智能软件工厂设计建议  
> 参考项目：garrytan/gstack  
> 目标：判断 gstack 对“智能软件工厂 / agent-team”的可借鉴点、不可照搬点和落地路径。

---

## 1. 结论摘要

gstack 与智能软件工厂有相似的愿景：都试图把 AI 编码工具从“一个聊天框/一个 Copilot”提升为“有角色、有流程、有评审、有交付纪律的虚拟软件团队”。

但二者层次不同：

```text
gstack = 面向个人开发者和 AI Coding Agent 的工作流 Skill Pack
智能软件工厂 = 面向团队化数字员工、项目、文档、决策、运行时与状态治理的平台型产品
```

因此，智能软件工厂不应照搬 gstack 的仓库结构或 slash command 产品形态，而应吸收它的核心方法：

```text
1. 角色化 Skill，而不是散乱 prompt。
2. 计划先审查，再开发。
3. 评审、QA、安全、发布、复盘都应产品化为显式流程。
4. Skill 要能跨 Agent Host 分发。
5. 工作流要能生成可执行任务，而不是只输出建议。
6. 多模型 / 多智能体二次审查有价值。
7. 安全护栏和编辑边界必须是一等设计。
8. 上下文保存、恢复和长期记忆是长周期工程的关键。
```

## 2. gstack 当前形态

### 2.1 产品定位

gstack 自称是一套开源软件工厂，把 Claude Code 等 AI coding agent 组织成虚拟工程团队。其 README 描述了 CEO、工程经理、设计师、Reviewer、QA、安全官、Release Engineer 等多个角色，并强调这些能力通过 Markdown skill 和 slash command 使用。

核心特征：

```text
- 不是一个 Web 平台。
- 不是一个后端调度系统。
- 不是一个数据库驱动的多用户产品。
- 它主要是一组高度结构化的 SKILL.md + 脚本 + 工作流约束。
```

### 2.2 Skill 体系

gstack 的 AGENTS.md 明确：gstack 是一组 SKILL.md 文件，每个 Skill 是一个专门角色，例如 CEO reviewer、eng manager、designer、QA lead、release engineer、debugger 等。

它的 Skill 不只是“提示词”，而是带有：

```text
- 触发命令
- 角色定位
- 使用时机
- 输入输出格式
- 决策问题格式
- 工具调用策略
- 安全约束
- 任务产物要求
```

这对智能软件工厂的启发是：SkillVersion 不能只保存一段 prompt 文本，至少要支持 Markdown 正文、脚本、资源、元信息、适用岗位、验收方式和运行时初始化规则。

### 2.3 工作流阶段

gstack 的工作流覆盖：

```text
想清楚 / 重新定义问题：/office-hours
计划审查：/plan-ceo-review、/plan-eng-review、/plan-design-review、/plan-devex-review
自动计划聚合：/autoplan
开发中审查：/review、/codex、/investigate
视觉与设计：/design-review、/design-shotgun、/design-html
QA：/qa、/qa-only、/browse
安全：/cso、/careful、/freeze、/guard
发布：/ship、/land-and-deploy、/canary
文档：/document-release、/document-generate
复盘与记忆：/retro、/learn、/context-save、/context-restore、/setup-gbrain、/sync-gbrain
```

这说明高评分项目的关键不是“AI 会写代码”，而是把软件交付的前后环节全部流程化。

## 3. 与智能软件工厂的相似点

| 维度 | gstack | 智能软件工厂 |
|---|---|---|
| 角色化 | Skill 扮演 CEO、Eng Manager、QA、Release 等角色 | AgentTemplate / Worker 扮演 Leader、架构、开发、测试、设计、建模等岗位 |
| Skill 资产 | SKILL.md + scripts + resources | Skill / SkillVersion / TemplateSkillMapping |
| 软件工厂心智 | 一个人借助多个 AI 角色完成软件交付 | 项目负责人管理多名数字员工完成项目交付 |
| 流程治理 | think → plan → build → review → test → ship → reflect | Skill → Template → Worker → Team/Project → Doc → Decision → Activity |
| 评审门禁 | plan review、code review、QA、安全、发布检查 | 待决策、文档发布、原型评审、未来多智能体评审 |
| 多 Agent Host | 支持 Claude Code、Codex CLI、OpenCode、Cursor 等 | 设计中已有 RuntimeHost / runtimeType / OpenCode runtime 抽象 |
| 运行时边界 | skill 安装到不同 agent 的 skill 目录 | WorkerRuntimeBinding 初始化技能快照到 workspaceDir |
| 上下文延续 | context-save / restore / gbrain | project-memory、handoff、ConversationRouteBinding、Activity |

## 4. 可直接借鉴的设计

### 4.1 Skill 不只是 prompt，而是“角色工作单元”

建议在智能软件工厂中把 SkillVersion 扩展为以下心智：

```text
SkillVersion = 指令正文 + 使用时机 + 输入要求 + 输出格式 + 工具权限 + 验收方式 + scripts/resources
```

P0a 可先只实现 Markdown 内容和元信息展示，但数据模型应预留：

```ts
type SkillVersion = {
  content: string
  usage?: string
  inputContract?: string
  outputContract?: string
  toolPolicy?: string
  verification?: string
  assets?: {
    scripts?: Array<{ path: string; description?: string }>
    resources?: Array<{ path: string; description?: string }>
  }
}
```

### 4.2 计划审查应成为一等流程

gstack 的 /plan-eng-review 聚焦 architecture、system boundaries、data flow、state transitions、failure modes、edge cases、trust boundaries、test coverage。

建议智能软件工厂引入“计划评审文档模板”，放在：

```text
docs/guides/智能软件工厂-计划评审模板-v0.6.33.md
```

每个 P0a / P0b / P1 任务执行前，OpenCode 应在任务说明中回答：

```text
- 本任务改哪些模块边界？
- 输入数据从哪里来？
- 状态如何流转？
- 错误和空状态是什么？
- 哪些路径不能触碰？
- 需要哪些最小测试或人工验收？
```

### 4.3 任务产物必须可执行

gstack 的 plan-eng-review 要求输出 Implementation Tasks，包含优先级、组件、文件、验证命令。

智能软件工厂也应把任务拆解从“描述”升级为“施工卡”：

```text
Task ID
目标
前置依赖
涉及目录
允许修改
禁止修改
数据契约
UI 验收
命令验收
回归风险
提交信息建议
```

### 4.4 Worktree 并行策略

gstack 的工程评审会分析 implementation steps 的并行机会，按模块目录而非具体文件判断冲突。

智能软件工厂后续多数字员工协同时，可采用：

```text
Lane A：contracts + domain，串行主干
Lane B：mock-seed，依赖 contracts
Lane C：UI primitives，可与 mock-seed 并行
Lane D：pages/features，依赖 contracts/domain/mock-seed/state
Lane E：chat-integration mock，可与页面并行，但不得接真实 Gateway
```

P0a 阶段建议仍以单主线为主；P0b/P1 再让多个 OpenCode worker 分 worktree 并行。

### 4.5 安全护栏产品化

gstack 的 /careful、/freeze、/guard 说明：AI 工程不是只要效率，还需要编辑边界和危险命令确认。

智能软件工厂中可以设计：

```text
RuntimePolicy
├── allowedWriteRoots
├── deniedWriteRoots
├── dangerousCommandPatterns
├── requireHumanApprovalPatterns
├── dryRunDefault
└── auditLevel
```

P0a 不执行命令，但可以在 Worker / RuntimeHost 详情中展示未来安全策略字段。

### 4.6 多模型二次审查

gstack 的 /codex 作为第二意见，可做 code review、adversarial challenge、open consultation。

智能软件工厂可以设计未来“Review Decision”类型：

```text
Decision.type = architecture_review | code_review | security_review | release_gate
Decision.requester = Leader / Reviewer Agent / Codex second-opinion
Decision.options = fix now / accept risk / split task / rollback
```

P0a 可先在 mock decision 中加入一条“架构评审待确认”示例。

### 4.7 文档发布后要自动更新文档

gstack 有 /document-release 和 /document-generate，用于防止 README、设计文档和已交付功能脱节。

智能软件工厂应把“文档同步”纳入 Activity 和任务验收：

```text
代码/原型变更完成后 → 生成文档更新建议 → 负责人确认 → 更新 docs → 写 Activity
```

P0a 可通过 Doc 发布和 Activity 回写模拟。

### 4.8 上下文保存与恢复

gstack 的 context-save / context-restore 和 gbrain 说明长周期开发需要跨会话记忆。

智能软件工厂已有 project-memory 和会话交接文档，建议固化为：

```text
docs/project-memory.md       # 长期事实
docs/handoffs/               # 会话交接
docs/changes/                # 版本变化
docs/reports/                # 评审与复盘
runtime snapshot             # P0b/P1 的 Worker 运行时快照
```

## 5. 不应照搬的部分

### 5.1 不照搬 slash command 作为产品主入口

gstack 面向 Claude Code / Codex / OpenCode 用户，slash command 是自然入口。

智能软件工厂面向负责人和数字员工管理，核心入口应是 Web 工作台：

```text
Skill 工作台
岗位模板
数字员工
团队项目
文档库
待决策
运行时监控
小云对话
```

slash command 可以成为运行时或开发者辅助能力，但不应成为产品主 UI。

### 5.2 不照搬“个人工作流”作为企业产品流程

gstack 很强是因为高度个人化、强观点、轻配置。智能软件工厂未来若面向企业，需要：

```text
权限
审计
多用户
可解释状态
可回滚文档
运行时隔离
团队协作记录
```

因此要保留平台治理模型，不要为了快速而去掉业务对象。

### 5.3 不在 P0a 引入真实浏览器 QA / 安全扫描 / 发布流水线

这些是 P1+ 的好方向，但 P0a 目标是前端可演示 MVP。提前引入会让范围失控。

### 5.4 不把 gstack memory / telemetry 设计直接引入

gstack 的记忆和遥测适合个人工具。智能软件工厂若引入长期记忆，必须先设计：

```text
数据归属
项目隔离
敏感信息扫描
访问控制
清理策略
```

## 7. 建议加入智能软件工厂产品路线的 gstack 风格能力

| 阶段 | 能力 | 对应 gstack 启发 | 是否 P0a |
|---|---|---|---|
| P0a | Skill / AgentTemplate / Mapping / RuntimeBinding | SKILL.md + role skills | 是 |
| P0a | OpenCode 执行清单 | Implementation Tasks | 是 |
| P0a | Activity 回写 | workflow trace | 是 |
| P0b | RuntimePolicy 安全策略 | careful / freeze / guard | 否 |
| P0b | 本地 runtime-service | host-specific skill install | 否 |
| P1 | 多智能体评审 | review + codex second opinion | 否 |

| P1+ | 长期记忆 / 知识库 | gbrain / context-save / context-restore | 否 |
