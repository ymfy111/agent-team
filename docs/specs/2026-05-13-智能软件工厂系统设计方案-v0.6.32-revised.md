# 智能软件工厂设计方案

> 状态：设计修订稿 v0.6.32
> 日期：2026-05-14
> 承接需求：`docs/specs/2026-05-13-智能软件工厂产品需求规格.md`  
> 原型基线：`docs/prototypes/Agent-Team-V2-注册中心-v0.6.32-ui-review-task-breakdown.html`  
> P0a 闭环演示原型：`docs/prototypes/Agent-Team-V2-注册中心-v0.6.32-p0a-flow.html`
> 设计目标：将原型与 PRD 转化为可实施的网站架构、模块边界、数据流、状态流转与集成方案。

## 1. 设计目标与范围

### 1.1 设计目标

本设计方案解决“如何把智能软件工厂原型实现为可持续演进的网站”的问题。设计重点包括：

1. 保持 v0.6.26-persona2 原型的信息架构、人物感和核心交互。
2. 将总览、团队、项目、员工、待决策、设置、团队工作台、项目文档库、小云对话拆成清晰模块。
3. 明确 Team、Worker、Project、Doc、Decision、Activity 等核心对象的数据关系。
4. 明确文档发布、决策处理、员工沟通、小云 target/session 等关键业务流。
5. 明确 agent-team 与 agent-web-kit 的边界，避免重复实现聊天 UI。

### 1.2 范围内

- 网站页面架构与路由组织。
- 前端业务模块边界。
- 核心数据模型与状态流转。
- 小云对话入口与 agent-web-kit 集成边界。
- 原型占位能力的设计处理策略。
- MVP 与后续迭代拆分建议。
- v0.6.32 技术落地补充：推荐 monorepo 工程结构、前后端技术路线、数据库演进路线和测试分层。

### 1.3 范围外

- 具体代码实现。
- 数据库物理表结构。
- 具体业务代码实现。
- 生产环境部署细节和完整运维方案。
- agent-web-kit 内部实现。
- OpenCode/Bridge 的底层协议细节实现。

## 2. 设计原则

### 2.1 原型保真

v0.6.26-persona2 是人物感与主交互基线。页面布局、导航心智、团队编队、项目文档库、小云入口应优先继承原型，不做无依据重构。

### 2.2 文档驱动

项目文档不是附件，而是 Agent 执行上下文。保存草稿只影响编辑状态；发布文档才代表将某个版本交给 Agent 读取、执行或参考。

### 2.3 Leader 作为沟通入口

普通成员不直接对话。员工卡若提供沟通入口，必须转译为“通过 Leader 联系”或切换到所属团队 Leader 的小云会话。

### 2.4 业务数据与聊天 UI 解耦

agent-team 维护团队、项目、文档、决策等业务数据；agent-web-kit 负责对话组件、消息渲染、附件、问题卡、决策卡和事件订阅。

### 2.5 占位能力显式化

原型中的 alert、静态 mock、未触发 overlay、残留字段不能被默认为已完成能力。实现计划必须单独标记并决定保留、删除或补完。

## 3. 总体架构

### 3.1 分层结构

```text
agent-team
├── 页面层
│   ├── 运行态：总览 / 团队 / 项目 / 待决策；配置态：岗位 / 员工 / 技能 / 设置
│   └── 团队详情：团队工作台 / 项目文档库
├── 业务组件层
│   ├── 团队卡、员工卡、项目表、决策列表、文档树、历史抽屉
│   └── 小云入口包装组件
├── 业务状态层
│   ├── teams / workers / projects view / decisions / docs
│   └── selectedTeam / selectedMember / selectedDoc / navOrigin / chatTarget
├── 数据访问层
│   ├── mock seed
│   ├── API client（后续）
│   └── browser storage adapter
├── 小云集成层
│   ├── target/session/context adapter
│   └── agent-web-kit widget/embed-sdk
└── 后端/持久化层（后续设计深化）
    ├── 业务数据 API
    ├── 文档版本 API
    └── 对话 Bridge / OpenCode fallback
```

### 3.2 系统架构图

以下架构图用于表达运行时边界：agent-team 负责业务编排与页面状态，agent-web-kit 负责对话组件和 Gateway 链路，OpenCode/Bridge 作为后续可接入的执行与会话底座。

|   |   +-- 团队详情: 工作台 / 项目文档库
|   |
|   +-- Business State
|   |   +-- teams / workers / decisions / docs
|   |   +-- selectedTeam / selectedMember / selectedDoc / chatTarget
|   |
|   +-- Integration Adapter
|       +-- 生成小云 target / session / context
|       +-- 接收 question / decision / message events
|
+-- embed / event contract
|
+-- agent-web-kit
|   |
|   +-- Widget / SDK
|   |   +-- 消息渲染 / 附件 / question / decision
|   |   +-- 错误状态 / 停止状态 / 等待回复状态
|   |
|   +-- Gateway
|       +-- 会话路由 / SSE 事件 / 模型或后端目标转发
|
+-- HTTP / SSE / bridge contract
|

    |

### 3.3 依赖方向

- 页面层依赖业务组件层和业务状态层。
- 业务组件通过事件回调改变状态，不直接访问持久化细节。
- 数据访问层对页面透明，后续可从 mock 平滑替换为 API。
- 小云集成层只接收业务上下文，不反向拥有团队/项目/文档数据。

### 3.4 v0.6.32 技术落地补充原则

本节补充从“原型与业务设计”走向“可开发工程”的推荐技术路线。该补充不改变 v0.6.32 产品范围，也不要求立即进入完整后端实现；它的目标是让 P0a 前端可演示 MVP、P0b 本地可运行 MVP、P1 服务化增强之间有清晰演进路径。

设计原则：

```text
P0a：先验证页面、状态和业务闭环，不被后端阻塞。
P0b：补本地 API、SQLite 持久化和文档/技能/决策版本管理。
P1：升级 PostgreSQL、多用户权限和更完整的运行时集成。
```

P0a 的唯一主线验收故事是：`Skill → AgentTemplate → Worker → Team/Project → Doc 发布 → Decision 处理 → Activity 回写`。P0a 页面、组件和 mock 数据均应服务这条主线；不服务该主线的复杂筛选、批量操作、真实权限、完整审计和真实运行时集成，均不进入 P0a 必做范围。

### 3.5 推荐 monorepo 工程结构

参考 agent-web-kit 的 pnpm workspace 分层思想，agent-team 推荐采用如下结构：

```text
agent-team/
├── apps/
│   ├── web                         # 前端应用：运行态 / 配置态 / 项目文档库
│   └── api                         # 后端 API：P0b 引入，本地可运行服务
│
├── packages/
│   ├── contracts                   # 共享类型、枚举、DTO、API 契约
│   ├── domain                      # 领域模型、状态流转、派生规则
│   ├── mock-seed                   # P0a mock 数据与初始化脚本
│   ├── data-client                 # 前端 API client / mock client 统一入口
│   ├── state                       # 前端状态管理、selectors、派生视图
│   ├── ui                          # 智能软件工厂业务 UI 组件
│   └── chat-integration            # 对 agent-web-kit 的非侵入式适配层
│
├── db/
│   ├── schema.prisma               # P0b 引入
│   ├── migrations/
│   └── seed/
│
├── docs/
└── package.json / pnpm-workspace.yaml
```

### 3.6 包职责与依赖方向

| 包 / 应用 | 职责 | 不承担 |
|---|---|---|
| `apps/web` | 页面路由、页面组装、用户交互入口 | 不直接写数据库，不直接调 OpenCode |
| `apps/api` | 业务 API、文档版本、决策回写、技能发布 | 不渲染 UI，不保存 widget 内部状态 |
| `packages/contracts` | 类型、DTO、枚举、API 请求响应结构 | 不写业务流程，不依赖 UI |
| `packages/domain` | Team、Worker、AgentTemplate、Skill、Doc、Decision 等领域规则 | 不依赖 React、Fastify、Prisma |
| `packages/mock-seed` | P0a mock 数据、演示用初始状态 | 不作为生产数据源 |
| `packages/data-client` | mock client 与 HTTP client 的统一门面 | 不写页面状态 |
| `packages/state` | 前端 store、selectors、派生字段 | 不直接访问 DOM |
| `packages/ui` | 业务组件库：卡片、表格、文档树、抽屉等 | 不拥有主数据 |
| `packages/chat-integration` | 构造 HostContext、打开 `<agent-widget>`、映射 interaction/Decision | 不维护 Team / Project / Doc 主数据；不判断是否允许发布、是否创建正式 Decision 等业务规则 |

依赖方向：

```text
contracts
  ↓
domain ──→ contracts
  ↓
mock-seed / data-client / state
  ↓
ui
  ↓
apps/web

apps/api ──→ contracts + domain + db
chat-integration ──→ contracts + domain + data-client + agent-web-kit embed-sdk
```

要求：下层包不依赖上层应用；领域层不依赖 UI、API 框架和数据库实现。`packages/chat-integration` 只做对话框适配和事件转译，不承担业务决策；是否创建 Decision、是否允许发布、是否可回滚等规则由 `packages/domain` 或 `apps/api` 判断。

### 3.7 前端技术路线

推荐：

```text
React + Vite + TypeScript
```

状态管理建议：

```text
P0a：Zustand 或轻量 store + localStorage / IndexedDB adapter
P0b：保留同一 store 接口，data-client 从 mock client 切换到 HTTP client
P1：按页面需要补 React Query / TanStack Query 管理服务端状态
```

前端分层建议：

```text
apps/web/src/
├── app/                 # 路由、布局、全局 provider
├── pages/               # 总览、团队、项目、待决策、岗位、员工、技能、设置
├── features/            # 文档发布、决策处理、员工入组、技能发布等业务功能
├── entities/            # Team / Worker / AgentTemplate / Skill / Doc / Decision 展示与状态适配
├── widgets/             # 页面级复合区块
└── shared/              # 基础 UI、工具函数、hooks
```

原型迁移优先级：

```text
1. 页面布局与导航分组
2. 配置态：岗位 / 员工 / 技能
3. 运行态：团队 / 项目 / 总览 / 待决策
4. 项目文档库：保存草稿 / 模拟发布
5. 小云 mock adapter
6. agent-web-kit 真实接入
```

### 3.8 后端技术路线

P0a 不要求真实后端。P0b 开始建议采用：

```text
Fastify + TypeScript + Prisma + SQLite
```

理由：

- Fastify 足够轻量，适合本地工具型服务和后续 Gateway 风格接口。
- Prisma 便于从 SQLite 平滑迁移到 PostgreSQL。
- SQLite 适合单机本地工作台、P0b 可运行 MVP 和快速验证。

P1 如进入多人协作、权限、多项目并发写入，再升级：

```text
Fastify / NestJS + Prisma + PostgreSQL
```

### 3.9 数据库与持久化路线

| 阶段 | 存储方式 | 说明 |
|---|---|---|
| P0a | mock seed + localStorage / IndexedDB | 验证页面和业务闭环 |
| P0b | SQLite + Prisma | 本地可运行，支持持久化文档版本、技能发布、决策回写 |
| P1 | PostgreSQL + Prisma | 支持多人协作、权限、审计和并发写入 |

P0b 首批必须持久化的对象：

```text
AgentTemplate
Worker / AgentInstance
Skill
SkillVersion
TemplateSkillMapping
Team
Project
ProjectAssignment
Doc
DocVersion
Decision
Activity
```

P0b 后置 / P1 预留对象：

```text
ConversationBinding
ConversationContextSnapshot
DecisionInteractionMapping
User / Organization / RolePermission
AuditLog
DocReview / DocConflict
```

说明：P0b 目标是单用户本地可运行，不以真实账号体系、多用户权限、完整审计和真实对话会话绑定为前置条件。`Activity` 是业务事件摘要，不替代完整 `AuditLog`；完整审计在 P1 引入。

### 3.10 小云集成层在工程中的位置

小云集成不应散落在页面中，应集中到：

```text
packages/chat-integration/
```

该包负责：

```text
- openGlobalAssistant()
- openTeamLeader(teamId)
- openWithWorkerContext(workerId)
- openWithDocContext(docId)
- openWithDecisionContext(decisionId)
- buildHostContext()
- mapInteractionToDecision()
- handleAgentEvent()
```

该包不负责：

```text
- 渲染完整聊天 UI
- 保存 Team / Project / Doc 主数据
- 直接调用 OpenCode Runtime
- 维护 agent-web-kit 内部状态
```

边界要求：`chat-integration` 是适配层，不是业务规则层。它可以把 `interaction.request` 转译为候选业务输入，但是否创建 `Decision`、是否允许回写 `Activity`、是否触发文档发布上下文变更，必须由 `packages/domain` 或 `apps/api` 判定。`agent-web-kit` 只产生对话事件，不成为 `Decision`、`DocVersion`、`Activity` 的业务真源。

## 4. 页面与路由设计

### 4.1 主页面

### 4.1.1 导航分组规则

左侧导航分为运行态和配置态两组：

```text
运行态
├── 总览
├── 团队
├── 项目
└── 待决策

软件工厂配置
├── 岗位      → 页面标题：数字员工模板（岗位）
├── 员工      → 数字员工实例管理
├── 技能      → Skill / SKILL.md / 岗位技能匹配
└── 设置      → 只读运行信息
```

设计约束：

- 运行态页面优先展示当前团队、项目、决策、动态和执行上下文，不直接承担能力资产编辑。
- 配置态页面负责数字员工实例、数字员工模板、技能资产和系统运行信息。
- “岗位”是 UI 短名；技术模型仍为 `AgentTemplate`，防止与项目职责 `projectRole` 混淆。
- 岗位页放在配置态首位，因为它是数字员工实例的创建模板；员工页虽然展示运行状态，但主职责是数字员工实例管理，因此放在配置态。


| 路由/页面 | 职责 |
|---|---|
| 总览 | 全局指标、协作拓扑、团队动态、待分配池。 |
| 团队 | 团队卡列表，进入团队详情。 |
| 项目 | 从团队当前项目聚合的项目索引。 |
| 员工 | 全员检索、筛选、分组、加入团队。 |
| 待决策 | 决策列表、过滤、详情与处理。 |
| 岗位 | 数字员工模板（岗位模板）、职责、实例统计、岗位技能摘要、创建数字员工入口。 |
| 技能 | Skill Markdown 能力包、岗位技能匹配、影响范围、启停。 |
| 设置 | 只读运行信息。 |

### 4.2 团队详情

团队详情包含两个稳定 tab：

1. `workbench`：团队工作台，左编队右详情。
2. `docs`：项目文档库，左文档树右文档详情。

项目页点击“文档”时，应直接进入对应团队详情的 `docs` tab，并定位到默认或指定文档。

### 4.3 导航来源态

从总览拓扑进入团队详情或员工详情时，记录 `navOrigin=overview`，展示“来自协作全景”的返回 chip。返回后清除来源态。

## 5. 模块边界设计

### 5.1 总览模块

输入：teams、workers、decisions、activities。  
输出：页面跳转、打开抽屉、打开小云、动态过滤。  
禁止职责：不直接修改团队成员或项目文档。

### 5.2 团队模块

输入：teams、currentProject 摘要。  
输出：进入团队详情、触发团队更多操作。  
禁止职责：不承载文档编辑和决策处理。

### 5.3 项目模块

输入：从 teams[].currentProject 派生的 projects view。  
输出：进入团队详情、打开代码仓、进入项目文档库。  
禁止职责：不维护独立 projects 主数据，直到产品升级为多项目模型。

### 5.4 员工模块

输入：teams[].members + workers pool。  
输出：搜索筛选、加入团队、查看监控、通过 Leader 联系。  
禁止职责：不直接启动普通成员会话。

### 5.5 决策模块

输入：decisions。  
输出：过滤、查看详情、处理决策、回写动态/上下文。  
禁止职责：不直接编辑项目文档正文，但可产生决策记录或关联文档。

### 5.6 文档库模块

输入：selectedTeam.currentProject.docs、文档内容、协作状态。  
输出：选择文档、编辑、保存草稿、发布、撤销、查看历史。  
禁止职责：不直接实现聊天 UI，只负责为 Agent 提供发布上下文。

### 5.7 小云入口模块

输入：target、session、业务上下文。  
输出：打开/关闭 widget、切换 target、发送消息、接收 question/interaction 事件。  
禁止职责：不维护 Team/Project/Doc 主数据。

### 5.8 数字员工模板管理模块

输入：全局 agentTemplates[]、skills[]、TemplateSkillMapping、workers[]、teams/projects view。  
输出：数字员工模板列表、模板详情、职责说明、实例统计、模板技能摘要、创建数字员工实例、跳转技能页并带模板过滤。  
交互形式：左侧配置区菜单“岗位” + 卡片/表格视图 + 详情区 + 创建数字员工向导。  
禁止职责：不直接编辑 Worker 运行状态；不保存 Skill 正文副本；不把项目职责当作模板主数据。

### 5.9 技能管理模块

输入：全局 skills[]、agentTemplates[]、TemplateSkillMapping。  
输出：Skill CRUD、模板-Skill 匹配编辑、影响范围计算、草稿/发布版本管理、变更广播。  
交互形式：左侧配置区菜单“技能” + Skill 配置工作台 + 技能资产 tab + 岗位匹配 tab。  
禁止职责：不直接操作 Worker 实例状态；Worker.skills[] 由状态层从 TemplateSkillMapping 派生。设置页不承载技能编辑。

## 6. 核心数据模型设计

### 6.1 Team

```text
Team
├── id
├── name
├── leaderId
├── memberIds[]
├── health
├── masterStatus
├── currentProject
└── activities[]
```

设计要求：Team 是项目承接单元；当前阶段一组同时只有一个 currentProject。

### 6.2 Worker / Member

```text
Worker
├── id
├── name
├── agentTemplateId
├── projectRole
├── status
├── teamId | null
├── currentTaskSummary
├── skills[]          → 派生自通用技能 + 模板专属技能
└── heartbeatTs
```

设计要求：Worker 在未入组时属于员工池；入组后成为 Team member。普通 member 不拥有直接聊天入口。同一 AgentTemplate 可创建多个 Worker 实例（技能相同，分配不同团队/项目，各自独立 OpenCode 进程）。

### 6.3 Project

```text
Project
├── id
├── name
├── stage
├── health
├── codeRepo
├── modelRepo
├── docs[]
└── blockers[]
```

设计要求：Project 当前由 Team 聚合而来；项目页是视图，不是独立主数据源。

### 6.4 Doc

```text
Doc
├── id
├── category
├── title
├── status
├── version
├── authorId
├── reviewerIds[]
├── draftContent
├── publishedContent
└── updatedTs
```

设计要求：草稿与已发布版本语义分离。Agent 默认读取 publishedContent 或最新发布版本。

### 6.5 Decision

```text
Decision
├── id
├── teamId
├── projectId
├── requesterId
├── sourceType              → leader | doc_conflict | runtime_risk | interaction | manual
├── sourceInteractionId?    → 来自小云 interaction 时记录，用于幂等映射
├── type
├── title
├── status
├── context
├── options[]
├── resolution
├── handledBy?              → P0b 起记录处理人
├── handledAt?              → P0b 起记录处理时间
├── createdAt
└── expiresAt
```

状态流转：`pending → resolved | blocked | expired`。resolved 后应记录 `resolution、handledBy、handledAt` 并清除待处理提醒。

设计要求：Decision 是人在环控制链的业务主数据。P0a 可用 mock 状态模拟；P0b 起至少持久化处理结果、处理人和处理时间；完整 `auditTrail[]`、审批链、撤销与重放可放入 P1。

### 6.6 Activity

Activity 是团队动态和执行上下文的统一事件摘要。文档发布、决策处理、员工加入团队、小云关键事件都可生成 Activity。

### 6.7 AgentTemplate、Worker、Skill 与 TemplateSkillMapping

```text
AgentTemplate（数字员工模板）
├── id
├── code                  → explorer / fixer / oracle / designer
├── displayName
├── description
├── responsibilities[]
├── defaultRuntime
├── createPolicy
├── enabled
├── instanceCount         → 派生统计
└── onlineCount           → 派生统计

Worker / AgentInstance（数字员工实例）
├── id
├── name
├── agentTemplateId
├── projectRole           → 当前项目职责帽子，不等于模板主数据
├── teamId | null
├── projectId | null
├── status
├── currentTaskSummary
├── runtimeSession
├── skillVersionSnapshot? → 可选：创建或刷新时的技能版本快照
├── skills[]              → 派生自通用技能 + 模板匹配技能
└── heartbeatTs

Skill（Markdown 能力资产）
├── id
├── name
├── description
├── version
├── status                → draft / published / disabled
├── scope                 → global / template
├── frontmatter
├── markdownContent       → SKILL.md 正文
├── files[]               → SKILL.md / scripts / resources
├── publishedVersion
├── createdAt
└── updatedAt

TemplateSkillMapping（模板-Skill 多对多关联，唯一主数据）
├── agentTemplateId
├── skillId
├── source                → global / template
├── enabled
└── effectiveVersion      → 可选：锁定或默认跟随最新发布版本
```

设计要求：

- AgentTemplate 是数字员工创建模板；Worker 是由 AgentTemplate 实例化出来的运行实体。
- Worker 继承 AgentTemplate 的生效技能集，可被分配到不同团队和项目；MVP 不支持单实例覆盖模板技能。
- Skill 是独立可复用的 Markdown 能力资产，不从属于某个 AgentTemplate；核心内容是 `SKILL.md`。
- scope=global 的技能为通用技能，对全部数字员工模板生效。
- scope=template 的 Skill 通过 TemplateSkillMapping 匹配数字员工模板；一个 Skill 可匹配多个模板，一个模板可挂多个 Skill。
- Worker.skills[] 是运行时派生视图：= 通用技能 + TemplateSkillMapping[worker.agentTemplateId] 匹配的已发布技能版本。
- 修改 Skill 草稿不影响实例；发布 Skill 后才影响引用该 Skill 的数字员工模板和数字员工实例。
- 岗位管理入口在左侧配置区菜单“岗位”，页面标题显示“数字员工模板（岗位）”。
- 技能管理入口在左侧配置区菜单“技能”，支持：Skill 配置工作台、技能资产列表、岗位技能匹配。
- 保存/发布 Skill 时显示影响范围提示（影响 N 个数字员工模板、M 个数字员工实例）。


### 6.8 Skill 文件资源存储策略

Skill 的核心内容是 `SKILL.md`，后续可能包含 `scripts/` 与 `resources/`。不同阶段的资源策略如下：

| 阶段 | SKILL.md | scripts / resources |
|---|---|---|
| P0a | mock seed / 前端状态模拟 | 保留文件树与引用关系，不要求真实文件写入 |
| P0b | DB Text 或本地文件正文 + DB 元数据 | 本地文件系统路径 + DB 元数据，避免把二进制资源直接塞入业务表 |
| P1 | 可升级为对象存储、Git 式版本库或独立资源服务 | 支持多人协作、权限、版本差异和回滚 |

设计要求：Skill 资源的业务真源是技能版本，而不是 Worker 实例；`Worker.skills[]` 只引用已发布技能版本的派生视图。

## 7. 状态管理设计

### 7.1 全局业务状态

全局状态包含 teams、workers、decisions、roles、skills、TemplateSkillMapping，以及从 teams 派生的 projects view。实现初期可从 mock seed 初始化，后续替换为 API。Worker.skills[] 由状态层从 skills + TemplateSkillMapping 实时派生，不独立存储。

### 7.2 页面局部状态

- 总览：动态过滤团队、hover 团队。
- 团队详情：selectedTeam、activeTab、selectedMember、selectedDoc。
- 员工页：keyword、roleFilter、statusFilter、groupMode、collapsedGroups。
- 岗位页：selectedTemplate、templateKeyword。
- 技能页：activeSkillTab、skillKeyword、selectedRoleForMapping、selectedSkill。
- 待决策页：statusFilter、teamFilter、urgencyFilter、selectedDecision。

### 7.3 小云状态

小云状态包含 targetId、sessionId、mode、connectionStatus、windowMode、attachmentDraft、pendingInteraction。target 切换必须同步加载对应 runtime config 和历史。

## 8. 关键业务流设计

### 8.1 总览进入团队详情

用户点击团队/Leader → 设置 selectedTeam 与 navOrigin → 进入团队详情 workbench → 默认选中 Leader。

### 8.2 项目进入文档库

用户点击项目页文档入口 → 根据项目找到承接团队 → 进入团队详情 docs tab → 默认选中最近更新或优先级最高文档。

### 8.3 员工加入团队

用户在员工页选择未入组员工 → 选择目标团队 → 更新 worker.teamId 与 team.memberIds → 总览、团队详情、员工页同步刷新 → 生成团队动态。

### 8.4 通过 Leader 联系员工

用户在员工卡点击沟通 → 系统提示“通过所属团队 Leader 联系” → 打开小云并切换到该团队 Leader target → 注入员工上下文。

### 8.5 文档发布

用户编辑文档 → 保存草稿 → 发布给 Agent → 生成 published version → 更新文档状态/协作信息 → 生成团队动态 → 小云/Agent 后续读取该版本作为上下文。

### 8.6 基于数字员工模板创建数字员工

用户在模板页选择数字员工模板 → 点击“创建数字员工” → 填写实例名称、目标团队、匹配项目和运行参数 → 系统写入 Worker.agentTemplateId / teamId / projectId → 根据 TemplateSkillMapping 派生 Worker.skills[] → 总览、员工页、团队详情同步刷新 → 生成员工创建 Activity。

### 8.7 数字员工模板与 Skill 变更

用户在技能页编辑 `SKILL.md` → 保存草稿 → 校验 YAML/frontmatter 与 Markdown 正文 → 点击发布生效 → 计算影响数字员工模板和实例数 → 更新 skills.publishedVersion / TemplateSkillMapping → 重新派生 Worker.skills[] → 刷新员工页、模板页、员工详情技能摘要 → 生成配置变更 Activity。

### 8.8 决策处理

用户打开待决策详情 → 选择方案并确认 → 写入 resolution → 状态改为 resolved → 更新提醒数量 → 生成团队动态 → 关联文档/项目上下文可追踪。

## 9. 团队与员工设计

### 9.1 团队卡

团队卡展示团队名、Leader、健康状态、在线人数、当前任务、当前项目、文档完成度和仓库状态。点击卡片进入团队详情。

### 9.2 编队列表

团队详情左侧编队只做选择与状态速览。Leader 轻微强调；成员突出头像、姓名、项目职责、状态、当前关注点。

### 9.3 员工监控

监控入口展示员工状态、当前关注点、最近活动、所属项目职责、心跳时间。监控不等于直接指挥。

### 9.4 更多操作

更多操作分阶段处理：MVP 可保留为不可用或弱提示；增强版再加入查看档案、调整职责、移出团队、转组等动作。

## 10. 项目与文档库设计

### 10.1 项目索引

项目页从 teams[].currentProject 派生，避免维护双主数据。若未来支持一组多项目，再升级为独立 projects[]。

### 10.2 文档树

文档按 specs、plans、decisions、reports、notes 分类。每个文档显示状态、版本、最近更新时间和协作提示。

### 10.3 编辑与发布

保存草稿不影响 Agent 上下文。发布会生成正式版本，并可被 Agent 读取。撤销发布不删除历史，只撤销本次发布生效。

### 10.4 历史抽屉

历史抽屉用于解释文档变更因果链。历史项应区分普通编辑、决策驱动、阻塞处理、发布、撤销、回滚。

## 11. 决策系统设计

### 11.1 决策来源

决策可来自 Leader、小云 interaction、文档冲突、不可逆操作、需求歧义、偏好选择等。

### 11.2 状态流转

```text
pending ──确认方案──> resolved
pending ──无法处理──> blocked
pending ──超时──────> expired
blocked ──补充信息──> pending
```

### 11.3 回写机制

决策处理后应至少回写团队动态。若决策关联文档，则文档历史应记录该决策；若决策来自小云，则小云会话中应能看到处理结果。

## 12. 小云对话集成设计

### 12.1 agent-team 职责

- 提供打开小云的入口。
- 选择 target：全局助手或团队 Leader。
- 传入业务上下文：team、project、doc、decision、worker。
- 接收 question/interaction/decision 事件后映射为业务动作。

### 12.2 agent-web-kit 职责

- 渲染对话 UI。
- 支持文本、图片、富文本、代码块、状态 callout。
- 支持问题卡、决策卡、多问题 wizard。
- 管理 widget/embed-sdk、SSE 订阅、Bridge/OpenCode adapter。

### 12.3 target 与 session

统一使用 session 术语。target 表示对话对象，例如 global assistant 或 team leader；session 表示该 target 下的具体会话上下文。

## 13. agent-web-kit 参考边界

### 13.1 应参考

- widget/embed-sdk 接入方式。
- session 事件模型。
- 消息渲染协议。
- question/interaction 机制。
- 附件展示和发送模型。

### 13.2 不应复制

- agent-team 的页面信息架构。
- 团队、项目、员工、文档、决策业务模型。
- 权限和组织关系判断。
- 项目文档库和决策中心的业务状态。

## 14. 数据来源与持久化策略

本节统一 P0a / P0b / P1 阶段定义，后续 PRD、设计、集成方案、开发计划和任务拆解均以此为准。

### 14.1 P0a：前端可演示 MVP

目标：验证页面、状态、mock 数据和端到端业务故事，不被后端、数据库、真实 Gateway 或 OpenCode Runtime 阻塞。

```text
数据来源：packages/mock-seed
状态管理：packages/state
本地缓存：localStorage / IndexedDB
API client：packages/data-client/mockClient
小云：MockChatAdapter
```

P0a 可以本地模拟：

- 岗位模板创建与编辑。
- 基于岗位模板创建数字员工实例。
- 技能保存草稿与发布生效。
- 员工加入团队和项目匹配。
- 文档保存草稿与发布给 Agent。
- 决策创建、处理和团队动态回写。
- 小云 mock adapter 的 question / decision 交互。

P0a 不要求：

- 真实后端服务。
- 真实数据库迁移。
- 真实 OpenCode Runtime 会话。
- 真实 agent-web-kit / agent-gateway 链路。
- 多用户权限和完整审计。

### 14.2 P0b：本地可运行 MVP

目标：引入真实 API 和本地数据库，让核心业务对象可以持久化，并让文档版本、技能版本、决策处理结果可恢复。

推荐组合：

```text
Fastify + Prisma + SQLite
```

P0b 首批必须持久化：

| 对象 | 说明 |
|---|---|
| AgentTemplate | 岗位 / 数字员工模板 |
| Worker | 数字员工实例 |
| Skill | 技能资产元信息 |
| SkillVersion | SKILL.md 发布版本 |
| TemplateSkillMapping | 模板与技能的唯一匹配主数据 |
| Team | 团队与成员关系 |
| Project / ProjectAssignment | 当前项目视图与员工项目匹配 |
| Doc / DocVersion | 草稿、发布版本与历史 |
| Decision | 决策请求、状态、处理结果、处理人和处理时间 |
| Activity | 团队动态与事件摘要 |

P0b 后置 / P1 预留：

| 对象 | 说明 |
|---|---|
| ConversationBinding | target 与 conversationId / runtimeSessionId 绑定；真实 agent-web-kit 接入前不阻塞 P0b 首批 |
| ConversationContextSnapshot | 会话上下文快照；P0b 可由 agent-team 侧保存，P1 再与 Gateway 联动 |
| DecisionInteractionMapping | 小云 interaction 与 Decision 的持久映射；完整闭环放入 P1 |

P0b API 范围：

| API 域 | 能力 |
|---|---|
| AgentTemplate API | 岗位模板 CRUD、启停、模板详情 |
| Skill API | 技能草稿、技能版本、发布生效、影响范围 |
| Worker API | 数字员工实例创建、入组、项目匹配、状态更新 |
| Team API | 团队列表、团队详情、成员关系、团队动态 |
| Project API | 当前项目视图、项目匹配、项目文档库入口 |
| Doc API | 草稿、发布版本、历史、撤销发布 |
| Decision API | 决策列表、详情、处理、过期、回写 |
| Activity API | 团队动态、事件摘要 |

### 14.3 P1：服务化与真实集成增强

目标：升级多人协作、权限、审计和真实运行时集成能力。

当系统需要多人协作、权限、并发写入和更稳定的审计能力时，升级到：

```text
PostgreSQL + Prisma
```

P1 增强范围：

- 用户、组织、权限。
- 审计日志和关键动作追踪。
- 多项目并发。
- 文档审阅与冲突合并。
- agent-web-kit + agent-gateway + OpenCode 完整链路。
- 对话上下文快照和 `context.update` 协议增强。
- interaction 与 Decision 的持久映射和回写。
- 更完整的运行时状态和心跳。

### 14.4 主数据与派生数据规则

| 对象 | 主数据归属 | 说明 |
|---|---|---|
| AgentTemplate | 业务库 | 岗位 / 数字员工模板 |
| Skill | 业务库 + 版本表 | 技能资产，正文为 SKILL.md |
| TemplateSkillMapping | 业务库 | 模板与技能的唯一匹配主数据 |
| Worker | 业务库 | 数字员工实例 |
| Worker.skills[] | 派生视图 | 从模板技能匹配和技能版本计算 |
| Project | P0a/P0b 可从 Team.currentProject 派生 | 一团队多项目后升级为独立主数据 |
| DocVersion | 文档版本库 | 发布给 Agent 的上下文版本 |
| Decision | 业务库 | 人在环决策闭环 |
| Activity | 事件摘要 | 从业务动作生成，可持久化 |
| ConversationBinding | 会话索引 | P0b 可选 / P1 增强；target 与 conversationId/runtimeSessionId 绑定 |

### 14.5 一致性要求

1. 保存草稿不影响 Agent 上下文。
2. 发布文档生成新的 `DocVersion`，并更新可读发布版本。
3. 发布技能生成新的 `SkillVersion`，并刷新引用模板和实例的生效技能视图。
4. 处理决策必须写入 `Decision.resolution、handledBy、handledAt`，并生成 `Activity`。
5. 小云正式决策必须通过幂等键映射到唯一 Decision；真实映射表可在 P1 完整持久化。
6. 文档发布后的会话上下文更新分阶段处理：P0a 更新 MockChatAdapter，P0b 保存 agent-team 侧上下文快照，P1 再对接真实 `context.update`。
7. `Worker.skills[]` 不落库为主数据，只作为派生展示或缓存。

## 15. 错误、空状态与降级设计

| 场景 | 展示策略 | 恢复路径 |
|---|---|---|
| 无团队 | 空状态说明尚未创建团队 | 引导导入/创建团队（后续） |
| 无项目 | 团队卡显示未承接项目 | 引导分配项目（后续） |
| 无文档 | 文档库显示空分类 | 引导新建文档（后续） |
| 无决策 | 待决策页显示清空状态 | 返回总览 |
| 员工离线 | 状态置灰，显示离线时间 | 等待心跳恢复 |
| 文档冲突 | 显示冲突提示 | 进入协作/合并流程（后续） |
| 小云连接失败 | 显示连接失败与测试入口 | 切 mock 或重试连接 |
| 发布失败 | 保留草稿并提示失败 | 重试发布 |

## 16. 原型迁移与差异处理

### 16.1 可复用心智

- 8 个主导航页面按两组展示：运行态为总览、团队、项目、待决策；配置态为岗位、员工、技能、设置。
- 团队详情双 tab。
- 左编队右详情。
- 项目文档库的预览/编辑/发布模式。
- 小云浮层和弹窗心智。

### 16.2 需要补齐

- 岗位页/数字员工模板页职责和实例统计。
- 技能页从设置页拆出后的导航、路由和数据联动。
- 团队更多操作。
- 员工更多操作。
- 移除/转组规则。
- 决策提交真实闭环。
- 文档回滚机制。

## 17. 实施阶段划分

实施阶段统一使用 P0a / P0b / P1。核心原则：先验证产品闭环，再做本地可运行，最后做服务化与真实运行时集成。

### 17.1 P0a：前端可演示 MVP

核心目标：用 mock seed 和本地状态跑通最小业务闭环，验证运行态 / 配置态、岗位 / 员工 / 技能、文档发布和决策处理的产品心智。

P0a 范围：

- 导航与页面：运行态为总览、团队、项目、待决策；配置态为岗位、员工、技能、设置。
- 配置态三件套：岗位模板、数字员工实例、Skill 工作台。
- 文档驱动：文档库预览/编辑/保存草稿/模拟发布。
- 决策闭环：决策列表、过滤、详情、处理、状态流转、回写 Activity。
- 小云入口：保留入口、target 文案和 MockChatAdapter，不接真实 agent-web-kit。
- 总览 / 项目：作为读侧投影，展示 mock 数据和页面联动。

P0a 最小闭环故事：

```text
1. 查看或创建一个 Skill。
2. 将 Skill 挂到数字员工模板。
3. 基于模板创建数字员工实例。
4. 将数字员工加入团队并匹配项目。
5. 在团队 / 项目 / 总览中看到状态变化。
6. 编辑项目文档，保存草稿并发布。
7. 生成一条模拟决策请求。
8. 在待决策页处理该请求。
9. 团队动态出现对应 Activity。
```

P0a 不要求真实后端、真实数据库、真实 Gateway、真实 OpenCode Runtime、多用户权限和完整审计。

### 17.2 P0b：本地可运行 MVP

核心目标：引入本地 API 和 SQLite 持久化，让核心业务对象、文档版本、技能版本和决策处理结果可恢复。

P0b 范围：

- `apps/api`：Fastify 本地 API。
- `db/schema.prisma`：SQLite + Prisma 数据模型。
- 核心对象持久化：AgentTemplate、Worker、Skill、SkillVersion、TemplateSkillMapping、Team、Project、ProjectAssignment、Doc、DocVersion、Decision、Activity。
- Skill 发布、文档发布、决策处理具备持久化记录。
- 小云仍可使用 mock adapter；可选接入 agent-web-kit 文本对话，但不要求完整 interaction → Decision → 回写闭环。

P0b 不应被真实 Gateway / OpenCode 链路阻塞。

### 17.3 P1：服务化与真实集成增强

核心目标：完成连接能力和服务化增强，打通真实 agent-web-kit / agent-gateway / OpenCode 链路，并增强权限、审计和多人协作。

P1 范围：

- PostgreSQL + Prisma。
- 用户、组织、权限、审计日志。
- agent-web-kit 真实 `<agent-widget>` 接入。
- agent-gateway / OpenCode Runtime 真实会话。
- interaction.request → Decision 持久映射。
- 决策处理结果回写小云会话、文档历史和 Activity。
- 文档上下文 `context.update` 或等价协议。
- 文档冲突合并、审阅流、多项目并发等增强能力。
