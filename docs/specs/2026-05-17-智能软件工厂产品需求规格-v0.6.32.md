# 智能软件工厂产品需求规格

> 版本：v0.6.32  
> 日期：2026-05-17  
> 状态：当前产品正本 / AI 原生岗位与任务单闭环版

---

## 1. 产品定位

智能软件工厂是面向 AI 数字员工团队的软件研发协作系统。它将多个 AI 数字员工组织成团队，以项目、文档、任务单、决策和验收记录为工作载体，让负责人能够管理团队状态、追踪项目进展、处理关键决策，并通过团队 Leader 与数字员工团队协作。

当前 `agent-team` 项目本身就是智能软件工厂的半手工 POC：

```text
现在：
  项目负责人手动调度设计智能体、OpenCode / Codex 和评审会话协作。

未来：
  智能软件工厂通过岗位、项目、任务单、进度、阻塞、决策和验收门禁自动调度数字员工协作。
```

产品价值不是简单“让很多 Agent 聊天”，而是把 AI 软件研发过程中的岗位职责、上下文传递、任务分派、进度跟踪、质量门禁和人在环决策系统化。

## 2. 核心价值

### 2.1 从手动调度智能体到系统调度数字员工

没有智能软件工厂时，项目负责人需要反复告诉每个智能体：

```text
你是谁？
你负责什么？
你该读哪些文档？
这次任务是什么？
遇到问题找谁？
做到什么程度算完成？
谁负责验收？
```

有智能软件工厂后，这些信息由系统通过岗位、团队、项目、文档、任务单和状态流统一管理。

### 2.2 AI 原生岗位，不照搬人类岗位

智能软件工厂的岗位不是人类软件组织的机械映射，而是按大模型适合完成的工作闭环设计：

```text
协同规划岗：把用户意图变成文档、计划和任务单。
实现验证岗：把任务单变成可运行、可测试的产物。
交付审查岗：独立判断产物是否可以交付。
```

### 2.3 文档驱动 + 任务单驱动

文档解决“项目上下文是什么”，任务单解决“这次具体做什么”。

```text
需求 / 设计 / 计划 / 决策 / 验收记录
  是项目长期上下文。

WorkOrder / 任务单
  是数字员工本轮执行的具体合同。
```

### 2.4 Leader 作为沟通与协调入口

普通成员不直接面向用户沟通。团队 Leader 是用户和数字员工团队之间的统一入口，默认由协同规划岗担任。

Leader 负责：

```text
需求沟通
计划与任务单协调
进度跟踪
阻塞汇总
分歧初判
上报用户决策
反馈交付结果
```

## 3. AI 原生岗位与团队角色

### 3.1 岗位

| 岗位 | 说明 | 主要产物 |
|---|---|---|
| 协同规划岗 | 默认担任 Leader，负责用户沟通、需求沉淀、设计、计划、任务单、进度、阻塞和分歧初判。 | PRD、设计、计划、任务单、决策请求、进度汇总。 |
| 实现验证岗 | 负责编码、TDD、自测、修复和执行回执，可多个实例并行。 | 代码、测试、构建结果、执行回执、问题修复。 |
| 交付审查岗 | 负责独立验收、质量门禁、整改建议和初验收报工。 | 验收记录、整改意见、整改任务单、通过/不通过结论。 |

### 3.2 团队角色

| 团队角色 | 说明 |
|---|---|
| Leader / 组长 | 团队协调和用户沟通入口，默认由协同规划岗数字员工担任。 |
| Member / 成员 | 执行具体任务的数字员工，例如实现验证岗、交付审查岗。 |

岗位和团队角色是两个概念。一个协同规划岗 Worker 可以被任命为团队 Leader。

## 4. 核心对象

| 对象 | 说明 |
|---|---|
| Skill | 可复用能力资产。 |
| AgentTemplate | AI 原生岗位模板，例如协同规划岗、实现验证岗、交付审查岗。 |
| Worker | 数字员工实例，由岗位模板实例化。 |
| RuntimeHost | 承载 OpenCode / Browser 等运行时的节点。 |
| WorkerRuntimeBinding | Worker 与 OpenCode 独立进程 / workspace / 主智能体的绑定。 |
| Team | 数字员工团队，包含 Leader 和成员。 |
| Project | 团队当前承接的项目。 |
| Doc | 需求、设计、计划、任务、验收等项目文档。 |
| WorkOrder | 可派发、可执行、可验收的任务单。 |
| WorkOrderProgress | 任务进度状态。 |
| Blocker | 执行阻塞。 |
| Decision | 需要人类处理的决策项。 |
| ChangeRequest | 需求、设计、范围或任务变更请求。 |
| AcceptanceRecord | 交付审查记录。 |
| Activity | 团队动态和执行事件回写。 |

## 5. 页面需求

### 5.1 总览页

应展示：

```text
- 活跃团队数
- 在线数字员工数
- 进行中任务单数
- 阻塞数
- 待验收数
- 待决策数
```

总览页应直观看到每个团队的岗位编队：

```text
规划1 / 协同规划岗 / Leader
实现1-1 / 实现验证岗
实现1-2 / 实现验证岗
审查1 / 交付审查岗
```

### 5.2 团队工作台

团队工作台应突出 Leader 的协调职责：

```text
- 当前阶段
- 当前任务单
- 任务进度
- 阻塞和分歧
- 待用户决策
- 待交付审查
- 最近活动
```

### 5.3 项目文档库

文档类型应包含：

```text
需求文档
系统设计
开发计划
任务单
变更请求
验收记录
整改意见
```

文档发布是改变数字员工执行上下文的关键动作。

### 5.4 待决策页

待决策页不仅承接需求决策，也承接执行过程中的分歧、范围变更、技术路线取舍和交付风险。

决策来源包括：

```text
用户需求不清
Leader 无法裁决
实现路径分歧
交付审查不通过
范围或优先级变化
不可逆操作
```

### 5.5 员工页

员工页应按 AI 原生岗位展示数字员工，而不是传统“架构/开发/测试/设计/建模”岗位。

筛选维度：

```text
协同规划岗
实现验证岗
交付审查岗
Leader / Member
在线 / 忙碌 / 阻塞 / 待验收
```

## 6. P0a 范围

P0a 只做前端可演示 Mock 闭环：

```text
- AI 原生岗位展示
- Team / Leader / Worker 展示
- WorkOrder 任务单展示
- Progress / Blocker / Decision / AcceptanceRecord mock 流转
- 文档发布占位
- 任务派发占位
- 决策处理占位
- 交付审查占位
- Activity 回写
```

P0a 不做：

```text
- 真实后端
- 真实数据库
- 真实 Gateway
- 真实 OpenCode Runtime
- 真实 agent-browser 调用
- 真实权限系统
```

---

## 7. 验收条件

P0a 原型和后续前端应至少能演示：

```text
1. 一个团队由协同规划岗 Leader、多个实现验证岗和交付审查岗组成。
2. Leader 能看到任务单进度、阻塞、待决策、待验收。
3. 实现验证岗的任务状态从 assigned/running/submitted 流转。
4. 交付审查岗能产生 passed / rework_required 结论。
5. 验收不通过能生成整改任务单。
6. Leader 无法裁决的问题能进入待决策页。
7. 所有关键动作生成 Activity。
```
