# 智能软件工厂设计方案

> 状态：设计修订稿 v0.6.28  
> 日期：2026-05-13  
> 承接需求：`docs/specs/2026-05-13-智能软件工厂产品需求规格.md`  
> 原型基线：`docs/prototypes/Agent-Team-V2-注册中心-v0.6.28-skill-role.html`  
> 设计目标：将原型与 PRD 转化为可实施的网站架构、模块边界、数据流、状态流转与集成方案。

## 1. 设计目标与范围

### 1.1 设计目标

本设计方案解决“如何把智能软件工厂原型实现为可持续演进的网站”的问题。设计重点包括：

1. 保持 v0.6.26-persona2 原型的信息架构、人物感和核心交互。
2. 将总览、团队、项目、员工、待决策、设置、团队工作台、项目文档库、小云对话拆成清晰模块。
3. 明确 Team、Worker、Project、Doc、Decision、Activity 等核心对象的数据关系。
4. 明确文档发布、决策处理、员工沟通、小云 target/session 等关键业务流。
5. 明确 agent-team 与 agent-web-kit 的边界，避免重复实现聊天 UI。

### 1.2 范围内

- 网站页面架构与路由组织。
- 前端业务模块边界。
- 核心数据模型与状态流转。
- 小云对话入口与 agent-web-kit 集成边界。
- 原型占位能力的设计处理策略。
- MVP 与后续迭代拆分建议。

### 1.3 范围外

- 具体代码实现。
- 数据库物理表结构。
- 具体后端框架选型。
- agent-web-kit 内部实现。
- OpenCode/Bridge 的底层协议细节实现。

## 2. 设计原则

### 2.1 原型保真

v0.6.26-persona2 是人物感与主交互基线。页面布局、导航心智、团队编队、项目文档库、小云入口应优先继承原型，不做无依据重构。

### 2.2 文档驱动

项目文档不是附件，而是 Agent 执行上下文。保存草稿只影响编辑状态；发布文档才代表将某个版本交给 Agent 读取、执行或参考。

### 2.3 Leader 作为沟通入口

普通成员不直接对话。员工卡若提供沟通入口，必须转译为“通过 Leader 联系”或切换到所属团队 Leader 的小云会话。

### 2.4 业务数据与聊天 UI 解耦

agent-team 维护团队、项目、文档、决策等业务数据；agent-web-kit 负责对话组件、消息渲染、附件、问题卡、决策卡和事件订阅。

### 2.5 占位能力显式化

原型中的 alert、静态 mock、未触发 overlay、残留字段不能被默认为已完成能力。实现计划必须单独标记并决定保留、删除或补完。

## 3. 总体架构

### 3.1 分层结构

```text
agent-team
├── 页面层
│   ├── 总览 / 团队 / 项目 / 员工 / 待决策 / 岗位 / 技能 / 设置
│   └── 团队详情：团队工作台 / 项目文档库
├── 业务组件层
│   ├── 团队卡、员工卡、项目表、决策列表、文档树、历史抽屉
│   └── 小云入口包装组件
├── 业务状态层
│   ├── teams / workers / projects view / decisions / docs
│   └── selectedTeam / selectedMember / selectedDoc / navOrigin / chatTarget
├── 数据访问层
│   ├── mock seed
│   ├── API client（后续）
│   └── browser storage adapter
├── 小云集成层
│   ├── target/session/context adapter
│   └── agent-web-kit widget/embed-sdk
└── 后端/持久化层（后续设计深化）
    ├── 业务数据 API
    ├── 文档版本 API
    └── 对话 Bridge / OpenCode fallback
```

### 3.2 系统架构图

以下架构图用于表达运行时边界：agent-team 负责业务编排与页面状态，agent-web-kit 负责对话组件和 Gateway 链路，OpenCode/Bridge 作为后续可接入的执行与会话底座。

```text
Browser / Web UI
|
+-- agent-team
|   |
|   +-- Pages
|   |   +-- 总览 / 团队 / 项目 / 员工 / 待决策 / 岗位 / 技能 / 设置
|   |   +-- 团队详情: 工作台 / 项目文档库
|   |
|   +-- Business State
|   |   +-- teams / workers / decisions / docs
|   |   +-- selectedTeam / selectedMember / selectedDoc / chatTarget
|   |
|   +-- Integration Adapter
|       +-- 生成小云 target / session / context
|       +-- 接收 question / decision / message events
|
+-- embed / event contract
|
+-- agent-web-kit
|   |
|   +-- Widget / SDK
|   |   +-- 消息渲染 / 附件 / question / decision
|   |   +-- 错误状态 / 停止状态 / 等待回复状态
|   |
|   +-- Gateway
|       +-- 会话路由 / SSE 事件 / 模型或后端目标转发
|
+-- HTTP / SSE / bridge contract
|
+-- Backend / Runtime（后续）
    |
    +-- Business Data API: 团队 / 员工 / 项目 / 决策 / 动态
    +-- Document Version API: 草稿 / 发布版本 / 历史 / 回滚
    +-- Conversation Bridge: 小云会话 / OpenCode session / 工具调用结果
    +-- Storage: 业务库 / 文档版本库 / 会话索引
```

### 3.3 依赖方向

- 页面层依赖业务组件层和业务状态层。
- 业务组件通过事件回调改变状态，不直接访问持久化细节。
- 数据访问层对页面透明，后续可从 mock 平滑替换为 API。
- 小云集成层只接收业务上下文，不反向拥有团队/项目/文档数据。

## 4. 页面与路由设计

### 4.1 主页面

| 路由/页面 | 职责 |
|---|---|
| 总览 | 全局指标、协作拓扑、团队动态、待分配池。 |
| 团队 | 团队卡列表，进入团队详情。 |
| 项目 | 从团队当前项目聚合的项目索引。 |
| 员工 | 全员检索、筛选、分组、加入团队。 |
| 待决策 | 决策列表、过滤、详情与处理。 |
| 岗位 | 岗位模板、职责、实例统计、岗位技能摘要。 |
| 技能 | 技能资产、岗位技能匹配、影响范围、启停。 |
| 设置 | 只读运行信息。 |

### 4.2 团队详情

团队详情包含两个稳定 tab：

1. `workbench`：团队工作台，左编队右详情。
2. `docs`：项目文档库，左文档树右文档详情。

项目页点击“文档”时，应直接进入对应团队详情的 `docs` tab，并定位到默认或指定文档。

### 4.3 导航来源态

从总览拓扑进入团队详情或员工详情时，记录 `navOrigin=overview`，展示“来自协作全景”的返回 chip。返回后清除来源态。

## 5. 模块边界设计

### 5.1 总览模块

输入：teams、workers、decisions、activities。  
输出：页面跳转、打开抽屉、打开小云、动态过滤。  
禁止职责：不直接修改团队成员或项目文档。

### 5.2 团队模块

输入：teams、currentProject 摘要。  
输出：进入团队详情、触发团队更多操作。  
禁止职责：不承载文档编辑和决策处理。

### 5.3 项目模块

输入：从 teams[].currentProject 派生的 projects view。  
输出：进入团队详情、打开代码仓、进入项目文档库。  
禁止职责：不维护独立 projects 主数据，直到产品升级为多项目模型。

### 5.4 员工模块

输入：teams[].members + workers pool。  
输出：搜索筛选、加入团队、查看监控、通过 Leader 联系。  
禁止职责：不直接启动普通成员会话。

### 5.5 决策模块

输入：decisions。  
输出：过滤、查看详情、处理决策、回写动态/上下文。  
禁止职责：不直接编辑项目文档正文，但可产生决策记录或关联文档。

### 5.6 文档库模块

输入：selectedTeam.currentProject.docs、文档内容、协作状态。  
输出：选择文档、编辑、保存草稿、发布、撤销、查看历史。  
禁止职责：不直接实现聊天 UI，只负责为 Agent 提供发布上下文。

### 5.7 小云入口模块

输入：target、session、业务上下文。  
输出：打开/关闭 widget、切换 target、发送消息、接收 question/interaction 事件。  
禁止职责：不维护 Team/Project/Doc 主数据。

### 5.8 岗位管理模块

输入：roles[]、workers、RoleSkillMapping、skills。  
输出：岗位模板列表、岗位详情、职责说明、实例统计、岗位技能摘要、创建数字员工实例、跳转技能页并带岗位过滤。  
交互形式：左侧一级菜单“岗位” + 卡片/表格视图 + 详情区。  
禁止职责：不直接编辑 Worker 运行状态；不保存技能正文副本。

### 5.9 技能管理模块

输入：全局 skills[]、roles[]、RoleSkillMapping。  
输出：技能 CRUD、岗位-技能匹配编辑、影响范围计算、变更广播。  
交互形式：左侧一级菜单“技能” + Skill 配置工作台 + 技能资产 tab + 岗位匹配 tab。  
禁止职责：不直接操作 Worker 实例状态；Worker.skills[] 由状态层从 mapping 派生。设置页不承载技能编辑。

## 6. 核心数据模型设计

### 6.1 Team

```text
Team
├── id
├── name
├── leaderId
├── memberIds[]
├── health
├── masterStatus
├── currentProject
└── activities[]
```

设计要求：Team 是项目承接单元；当前阶段一组同时只有一个 currentProject。

### 6.2 Worker / Member

```text
Worker
├── id
├── name
├── role
├── projectRole
├── status
├── teamId | null
├── currentTaskSummary
├── skills[]          → 派生自通用技能 + 岗位专属技能
└── heartbeatTs
```

设计要求：Worker 在未入组时属于员工池；入组后成为 Team member。普通 member 不拥有直接聊天入口。同一 role 可存在多个 Worker 实例（技能相同，分配不同团队/项目，各自独立 OpenCode 进程）。

### 6.3 Project

```text
Project
├── id
├── name
├── stage
├── health
├── codeRepo
├── modelRepo
├── docs[]
└── blockers[]
```

设计要求：Project 当前由 Team 聚合而来；项目页是视图，不是独立主数据源。

### 6.4 Doc

```text
Doc
├── id
├── category
├── title
├── status
├── version
├── authorId
├── reviewerIds[]
├── draftContent
├── publishedContent
└── updatedTs
```

设计要求：草稿与已发布版本语义分离。Agent 默认读取 publishedContent 或最新发布版本。

### 6.5 Decision

```text
Decision
├── id
├── teamId
├── projectId
├── requesterId
├── type
├── title
├── status
├── context
├── options[]
├── resolution
├── createdAt
└── expiresAt
```

状态流转：`pending → resolved | blocked | expired`。resolved 后应记录 resolution 并清除待处理提醒。

### 6.6 Activity

Activity 是团队动态和执行上下文的统一事件摘要。文档发布、决策处理、员工加入团队、小云关键事件都可生成 Activity。

### 6.7 RoleTemplate、Worker、Skill 与 RoleSkillMapping

```text
RoleTemplate（岗位模板 / 数字员工模板）
├── id                 → explorer / fixer / oracle / designer
├── code               → @explorer / @fixer / @oracle / @designer
├── displayName
├── description
├── responsibilities[]
├── defaultRuntime     → 默认模型、工具权限、启动参数、工作目录策略
├── createPolicy       → 是否允许创建多实例、默认命名规则、项目匹配规则
├── enabled
├── instanceCount      → 派生自 workers
└── updatedAt

Worker（数字员工实例）
├── id
├── name
├── roleTemplateId
├── teamId | null
├── projectId | null
├── runtimeSession
├── status
├── currentTaskSummary
├── skills[]           → 运行时派生，不持久化为主数据
└── heartbeatTs

Skill（独立 Markdown 能力资产）
├── id
├── name
├── description
├── version
├── status             → draft | published | disabled
├── scope              → global | role
├── frontmatter        → YAML 元数据，如 name/version/summary/trigger/roles
├── markdownContent    → SKILL.md 正文
├── files[]            → SKILL.md、scripts/、resources/ 等
├── publishedVersion
├── createdAt
└── updatedAt

RoleSkillMapping（岗位-技能多对多关联，唯一主数据）
├── roleTemplateId
├── skillId
├── source             → manual | preset | imported
├── enabled
└── updatedAt
```

设计要求：
- RoleTemplate 是岗位模板，也是数字员工创建模板；Worker 是由 RoleTemplate 实例化出来的运行实体。
- Worker 继承 RoleTemplate 的生效技能集，可被分配到不同团队和项目；MVP 不支持单实例覆盖岗位技能。
- Skill 是独立可复用的 Markdown 能力资产，不从属于某个 RoleTemplate；核心内容是 `SKILL.md`。
- scope=global 的技能为通用技能，对全部岗位模板生效。
- scope=role 的技能通过 RoleSkillMapping 匹配岗位模板；一个技能可匹配多个岗位模板，一个岗位模板可挂多个技能。
- Worker.skills[] 是运行时派生视图：= 通用技能 + RoleSkillMapping[worker.roleTemplateId] 匹配的已发布技能版本。
- 修改 Skill 草稿不影响实例；发布 Skill 后才影响引用该技能的岗位模板和数字员工实例。
- 技能管理入口在左侧一级菜单“技能”，支持：Skill 配置工作台、技能资产列表、岗位技能匹配。
- 岗位管理入口在左侧一级菜单“岗位”，展示岗位模板、创建数字员工入口、职责、实例统计和技能摘要。
- 保存/发布技能时显示影响范围提示（影响 N 个岗位模板、M 个数字员工实例）。

## 7. 状态管理设计

### 7.1 全局业务状态

全局状态包含 teams、workers、decisions、roles、skills、roleSkillMappings，以及从 teams 派生的 projects view。实现初期可从 mock seed 初始化，后续替换为 API。Worker.skills[] 由状态层从 skills + roleSkillMappings 实时派生，不独立存储。

### 7.2 页面局部状态

- 总览：动态过滤团队、hover 团队。
- 团队详情：selectedTeam、activeTab、selectedMember、selectedDoc。
- 员工页：keyword、roleFilter、statusFilter、groupMode、collapsedGroups。
- 岗位页：selectedRole、roleKeyword。
- 技能页：activeSkillTab、skillKeyword、selectedRoleForMapping、selectedSkill。
- 待决策页：statusFilter、teamFilter、urgencyFilter、selectedDecision。

### 7.3 小云状态

小云状态包含 targetId、sessionId、mode、connectionStatus、windowMode、attachmentDraft、pendingInteraction。target 切换必须同步加载对应 runtime config 和历史。

## 8. 关键业务流设计

### 8.1 总览进入团队详情

用户点击团队/Leader → 设置 selectedTeam 与 navOrigin → 进入团队详情 workbench → 默认选中 Leader。

### 8.2 项目进入文档库

用户点击项目页文档入口 → 根据项目找到承接团队 → 进入团队详情 docs tab → 默认选中最近更新或优先级最高文档。

### 8.3 员工加入团队

用户在员工页选择未入组员工 → 选择目标团队 → 更新 worker.teamId 与 team.memberIds → 总览、团队详情、员工页同步刷新 → 生成团队动态。

### 8.4 通过 Leader 联系员工

用户在员工卡点击沟通 → 系统提示“通过所属团队 Leader 联系” → 打开小云并切换到该团队 Leader target → 注入员工上下文。

### 8.5 文档发布

用户编辑文档 → 保存草稿 → 发布给 Agent → 生成 published version → 更新文档状态/协作信息 → 生成团队动态 → 小云/Agent 后续读取该版本作为上下文。

### 8.6 基于岗位模板创建数字员工

用户在岗位页选择岗位模板 → 点击“创建数字员工” → 填写实例名称、目标团队、匹配项目和运行参数 → 系统写入 Worker.roleTemplateId / teamId / projectId → 根据 RoleSkillMapping 派生 Worker.skills[] → 总览、员工页、团队详情同步刷新 → 生成员工创建 Activity。

### 8.7 岗位与技能变更

用户在技能页编辑 `SKILL.md` → 保存草稿 → 校验 YAML/frontmatter 与 Markdown 正文 → 点击发布生效 → 计算影响岗位模板和实例数 → 更新 skills.publishedVersion / roleSkillMappings → 重新派生 Worker.skills[] → 刷新员工页、岗位页、员工详情技能摘要 → 生成配置变更 Activity。

### 8.7 决策处理

用户打开待决策详情 → 选择方案并确认 → 写入 resolution → 状态改为 resolved → 更新提醒数量 → 生成团队动态 → 关联文档/项目上下文可追踪。

## 9. 团队与员工设计

### 9.1 团队卡

团队卡展示团队名、Leader、健康状态、在线人数、当前任务、当前项目、文档完成度和仓库状态。点击卡片进入团队详情。

### 9.2 编队列表

团队详情左侧编队只做选择与状态速览。Leader 轻微强调；成员突出头像、姓名、岗位、状态、当前关注点。

### 9.3 员工监控

监控入口展示员工状态、当前关注点、最近活动、所属项目职责、心跳时间。监控不等于直接指挥。

### 9.4 更多操作

更多操作分阶段处理：MVP 可保留为不可用或弱提示；增强版再加入查看档案、调整职责、移出团队、转组等动作。

## 10. 项目与文档库设计

### 10.1 项目索引

项目页从 teams[].currentProject 派生，避免维护双主数据。若未来支持一组多项目，再升级为独立 projects[]。

### 10.2 文档树

文档按 specs、plans、decisions、reports、notes 分类。每个文档显示状态、版本、最近更新时间和协作提示。

### 10.3 编辑与发布

保存草稿不影响 Agent 上下文。发布会生成正式版本，并可被 Agent 读取。撤销发布不删除历史，只撤销本次发布生效。

### 10.4 历史抽屉

历史抽屉用于解释文档变更因果链。历史项应区分普通编辑、决策驱动、阻塞处理、发布、撤销、回滚。

## 11. 决策系统设计

### 11.1 决策来源

决策可来自 Leader、小云 interaction、文档冲突、不可逆操作、需求歧义、偏好选择等。

### 11.2 状态流转

```text
pending ──确认方案──> resolved
pending ──无法处理──> blocked
pending ──超时──────> expired
blocked ──补充信息──> pending
```

### 11.3 回写机制

决策处理后应至少回写团队动态。若决策关联文档，则文档历史应记录该决策；若决策来自小云，则小云会话中应能看到处理结果。

## 12. 小云对话集成设计

### 12.1 agent-team 职责

- 提供打开小云的入口。
- 选择 target：全局助手或团队 Leader。
- 传入业务上下文：team、project、doc、decision、worker。
- 接收 question/interaction/decision 事件后映射为业务动作。

### 12.2 agent-web-kit 职责

- 渲染对话 UI。
- 支持文本、图片、富文本、代码块、状态 callout。
- 支持问题卡、决策卡、多问题 wizard。
- 管理 widget/embed-sdk、SSE 订阅、Bridge/OpenCode adapter。

### 12.3 target 与 session

统一使用 session 术语。target 表示对话对象，例如 global assistant 或 team leader；session 表示该 target 下的具体会话上下文。

## 13. agent-web-kit 参考边界

### 13.1 应参考

- widget/embed-sdk 接入方式。
- session 事件模型。
- 消息渲染协议。
- question/interaction 机制。
- 附件展示和发送模型。

### 13.2 不应复制

- agent-team 的页面信息架构。
- 团队、项目、员工、文档、决策业务模型。
- 权限和组织关系判断。
- 项目文档库和决策中心的业务状态。

## 14. 数据来源与持久化策略

### 14.1 MVP 阶段

- 使用 mock seed 初始化 teams、workers、decisions。
- 使用浏览器存储保存 UI 偏好、navOrigin、小云窗口状态、聊天历史缓存。
- 文档编辑可以先用本地状态模拟保存/发布。

### 14.2 可运行产品阶段

- Team、Worker、Project、Doc、Decision、Activity 进入后端 API。
- 文档版本和发布记录必须持久化。
- 小云 session 与业务 target 绑定关系必须持久化或可恢复。

### 14.3 外部链接

代码仓、模型仓是外部资源链接。agent-team 展示和跳转，不直接拥有仓库内容。

## 15. 错误、空状态与降级设计

| 场景 | 展示策略 | 恢复路径 |
|---|---|---|
| 无团队 | 空状态说明尚未创建团队 | 引导导入/创建团队（后续） |
| 无项目 | 团队卡显示未承接项目 | 引导分配项目（后续） |
| 无文档 | 文档库显示空分类 | 引导新建文档（后续） |
| 无决策 | 待决策页显示清空状态 | 返回总览 |
| 员工离线 | 状态置灰，显示离线时间 | 等待心跳恢复 |
| 文档冲突 | 显示冲突提示 | 进入协作/合并流程（后续） |
| 小云连接失败 | 显示连接失败与测试入口 | 切 mock 或重试连接 |
| 发布失败 | 保留草稿并提示失败 | 重试发布 |

## 17. 实施阶段划分

实施优先级按核心功能分层（见 PRD §2.4）排列：核心能力全部进 MVP，连接能力可先 mock，观察能力渐进补全。

### 17.1 MVP（P0 — 核心能力骨架）

核心目标：团队/员工可管理、文档可发布驱动 Agent、决策可闭环处理、技能可展示与管理。

- 团队与员工：团队卡列表、团队详情双 tab、编队列表、员工分配入组。
- 岗位管理：岗位列表/卡片 + 职责说明 + 实例统计 + 岗位技能摘要。
- 技能管理：左侧“技能”页 Skill 配置工作台（SKILL.md 编辑器）+ 技能资产列表 + 岗位技能匹配 + 员工卡技能摘要 + 员工详情技能列表。
- 文档驱动：文档库预览/编辑/保存草稿/发布（发布 = Agent 可读取）。
- 决策闭环：决策列表、过滤、详情、选择方案并确认、状态流转（pending → resolved/blocked/expired）、回写团队动态。
- 小云入口：占位组件 + mock adapter，不依赖 agent-web-kit 真实连接。
- 总览/项目：最简页面，展示 mock 数据，不承载核心交互。
- 员工页顶部统计条：员工总数、在线数、技能总数。
- 清理 librarian 残留和普通成员直接对话误导。

### 17.2 增强版（P1 — 连接能力打通）

核心目标：小云对话真实可用，文档版本完整，员工管理深化。

- 集成 agent-web-kit widget/embed-sdk，打通 Leader 对话。
- 接入真实业务 API 替换 mock seed。
- 文档版本、撤销发布、历史抽屉、回滚。
- 员工移除/转组、团队更多操作。
- 决策回写关联文档和小云会话。

### 17.3 后续迭代（P2 — 观察能力与扩展）

核心目标：总览仪表盘真实化，系统能力扩展。

- 总览统计指标真实化（替换静态 mock）。
- 协作拓扑动态数据。
- 一团队多项目。
- 团队暂停/重启/导出/历史。
- 更细权限体系。
- 文档冲突合并和审阅流。

## 18. 验收与测试策略

### 18.1 页面 smoke

验证 6 个主页面可进入、无空白、核心数据可见。

### 18.2 跨页面链路

验证总览到团队详情、项目到文档库、员工加入团队、待决策提醒跳转、小云 target 切换。

### 18.3 状态流转

验证员工入组、文档发布、决策处理、团队动态回写、提醒数量更新。

### 18.4 小云集成

验证打开关闭、target 识别、session 区分、文本/图片/问题卡/决策卡展示。

### 18.5 异常状态

验证无数据、离线、连接失败、发布失败、决策过期、文档冲突等状态不会造成页面不可用。
