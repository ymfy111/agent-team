## 1. 文档目的

本文把现有 Agent Team V2 注册中心原型沉淀为可阅读、可评审、可开发拆解的产品需求规格。本文同时服务两类读者：

- 产品/决策者：理解网站要解决什么问题、面向谁、提供哪些完整能力。
- 开发团队/AI 智能体：按页面和模块获得可执行的功能描述、交互要求和验收条件。

本文与原型共同构成需求基线：原型提供视觉与交互参照，本文提供结构化功能说明和验收标准。

## 2. 产品定位与价值主张

### 2.1 产品定位

智能软件工厂是一个智能体协作系统。它将多个 AI 智能体组织成研发团队，以项目和文档为工作载体，让负责人能够查看团队状态、追踪项目进展、处理决策请求，并与 Leader/主智能体进行对话。

### 2.2 目标用户

- 需要组建 AI 研发团队的项目负责人。
- 需要查看智能体工作状态、项目进度和阻塞项的技术 owner。
- 需要通过需求、计划、决策、报告等文档驱动 AI 团队开发的人。

### 2.3 核心价值

1. **团队化**：把 AI 智能体呈现为可识别的研发团队，而不是零散工具。
2. **项目化**：每个团队承接一个当前项目，项目包含代码仓、模型仓、文档库、阻塞项和阶段状态。
3. **文档驱动**：需求、计划、决策、报告、笔记成为团队协作的合同和上下文。
4. **可对话**：普通成员不直接对话；Leader/主智能体作为沟通入口，负责协调成员。
5. **可追踪**：总览、动态、决策、文档历史和员工状态共同形成可观察工作台。

### 2.4 核心功能分层

系统功能按"驱动力"分为三层：核心能力是系统存在的理由，连接能力是人机协作的通道，观察能力是核心能力的读侧投影。

```text
核心能力（没有这些系统不成立）
+-- 团队与员工管理
|   把 AI 智能体组织成有 Leader 的研发组
|   员工可分配、可监控、Leader 是唯一沟通入口
|
+-- 文档驱动执行
|   需求/计划/决策/报告作为 Agent 执行上下文
|   草稿 -> 发布 -> Agent 读取，发布是改变 Agent 行为的唯一动作
|
+-- 决策闭环
    Agent 遇到歧义/不可逆/偏好时上报决策请求
    人确认后回写执行，是人在环的关键控制点

连接能力（人与 Agent 的实时通道）
+-- 小云对话（通过 Leader）
    人与团队 Leader 对话，Leader 协调成员
    承载 question/decision 交互、上下文注入

观察能力（核心能力的读侧投影，不产生新动作）
+-- 总览仪表盘：全局指标、协作拓扑、团队动态
+-- 项目聚合视图：从团队当前项目派生，不维护双主数据

配置管理能力（软件工厂设置区，影响数字员工实例和运行时能力边界）
+-- 员工管理：查看和管理数字员工实例、分配团队/项目、观察运行状态
+-- 岗位管理：维护数字员工模板（岗位模板）的职责、实例统计、默认技能视图和创建入口
+-- 技能管理：维护 Skill Markdown 能力包、岗位技能匹配、启停和影响范围
+-- 设置页：只读运行信息，不承担配置编辑
```

**MVP 裁剪原则**：核心能力全部进入 MVP；连接能力可先用 mock adapter 占位；观察能力按简到繁渐进。

## 4. 信息架构

### 4.1 主导航与分区

左侧主导航分为两组，避免“运行态工作台”和“软件工厂配置”混在一起。

#### 4.1.1 运行态功能区

运行态功能区回答“当前团队和项目正在发生什么”。

| 页面 | 功能定位 |
|---|---|
| 总览 | 展示全局团队协作全景、指标、拓扑和动态流。 |
| 团队 | 展示团队列表，并进入团队详情，观察团队当前项目、编队和执行上下文。 |
| 项目 | 聚合展示各团队当前项目。 |
| 待决策 | 汇总需要负责人处理的决策请求，是运行过程中的人在环控制入口。 |

#### 4.1.2 软件工厂配置区

软件工厂配置区回答“先定义什么数字员工模板，再创建哪些数字员工实例，实例继承哪些技能，系统如何运行”。

| 页面 | 菜单显示 | 功能定位 |
|---|---|---|
| 数字员工模板 | 岗位 | 管理数字员工模板/岗位模板，维护职责、实例统计、默认技能视图和创建数字员工入口。 |
| 员工 | 员工 | 数字员工实例管理；展示全量实例、搜索筛选、加入团队、匹配项目和运行状态。 |
| 技能 | 技能 | 管理 Skill Markdown 能力包、岗位技能匹配、影响范围和启停状态。 |
| 设置 | 设置 | 只读展示系统运行信息、版本、端口、持久化路径和 checkpoint。 |

**命名规则**：菜单使用短名“岗位”，页面标题使用“数字员工模板（岗位）”。文档和代码使用 `AgentTemplate` 作为主模型名，“岗位模板”作为用户可理解的业务别名。

### 4.2 团队详情二级结构

团队详情稳定为双 tab：

1. **团队工作台**：左侧编队列表，右侧当前选中智能体/团队详情。
2. **项目文档库**：文档树、文档预览/编辑、协作信息、发布流程和历史抽屉。

### 4.3 小云对话入口

小云对话作为全局浮层能力存在，可从以下入口打开：

- 右下浮动按钮。
- 总览拓扑中的 Leader 对话入口。
- 团队详情中的 Leader 会话入口。
- 其他与 Leader/主智能体相关的操作入口。

## 5. 角色、团队与业务对象

### 5.1 组织模型

- 系统包含 5 个研发组。
- 每个研发组有 1 个 Leader 和若干成员。
- 一个团队同一时间承接 1 个当前项目。
- 未加入团队的员工作为“待分配/未认领员工”展示。
- **同一数字员工模板可创建多个数字员工实例**：例如 3 个 fixer 模板实例，继承相同技能，但分别分配到不同团队/项目，并各自运行独立的 OpenCode 进程。

### 5.2 角色体系

| 层级 | 角色/职责 | 说明 |
|---|---|---|
| 组织角色 | Leader / 组员 | Leader 是团队主控和对话入口；组员通过 Leader 协作。 |
| 模板类型 | explorer / fixer / oracle / designer | 当前有效数字员工模板类型；librarian 为历史残留，不进入新版本筛选和匹配。 |
| 项目职责 | 测试工程师 / 定制开发 / 架构师 / 设计师 / 建模师 | 用于表达成员在当前项目中的职责帽子。 |

### 5.3 数字员工模板与技能体系

#### 5.3.1 命名决策

为避免“岗位”同时被理解为 HR 岗位、项目职责或运行实例，本版本将配置入口统一命名为 **数字员工模板**。

| 层级 | 推荐名称 | 英文/代码名 | 说明 |
|---|---|---|---|
| 模板 | 数字员工模板 | AgentTemplate | 可实例化的数字员工能力模板，定义职责、默认技能、默认运行参数和创建策略。 |
| 实例 | 数字员工实例 | Worker / AgentInstance | 由数字员工模板创建的具体运行实体，可加入团队并匹配项目。 |
| 能力 | Skill / 技能 | Skill | Markdown 能力包，核心文件为 `SKILL.md`，可附带 scripts/resources。 |
| 关系 | 岗位技能匹配 | TemplateSkillMapping | 维护数字员工模板和 Skill 的多对多关系，是模板技能的唯一主数据。 |
| 项目职责 | 项目职责 / 项目任职 | projectRole / projectAssignment | 表达实例在某个项目中的职责帽子，不作为模板主数据。 |

**UI 命名规则**：左侧一级菜单使用短名“岗位”，页面标题使用“数字员工模板（岗位）”。文档和代码统一使用 `AgentTemplate`；历史文档中的 `RoleTemplate` / “岗位模板”仅作为兼容别名。

#### 5.3.2 数字员工模板模型

数字员工模板（AgentTemplate）是可创建数字员工实例的人员模板/能力模板。模板定义职责说明、默认运行参数、创建策略、默认技能视图和岗位技能匹配，但不保存技能正文副本。

| 概念 | 说明 |
|---|---|
| 数字员工模板 | 定义某一类数字员工的职责、默认运行参数、创建策略和默认能力边界。 |
| 数字员工实例 | 基于模板创建的具体 Worker，可加入团队并匹配项目。 |
| 模板技能 | 通过 TemplateSkillMapping 关联到模板的技能。 |
| 通用技能 | scope=global，对所有模板和实例生效。 |
| 项目匹配 | 实例创建或分配时选择目标团队/项目，决定其当前工作上下文。 |

#### 5.3.3 技能模型

技能（Skill）是独立的 Markdown 能力资产，通过**岗位技能匹配**关联到数字员工模板。一个 Skill 可匹配多个模板，一个模板可挂多个 Skill。

Skill 的核心内容是 `SKILL.md`，用于描述触发条件、操作流程、输入输出、质量检查和限制。Skill 可附带 `scripts/`、`resources/` 等文件，但 MVP 阶段只要求在原型中表达文件树和 Markdown 正文。

#### 5.3.4 关键规则

- 同一数字员工模板创建的所有数字员工实例默认共享相同的技能集：通用技能 + 模板匹配技能。
- 数字员工实例之间的差异主要在于实例名称、所属团队、匹配项目、当前任务、运行会话和运行状态，不在模板继承技能。
- 技能正文支持草稿与发布版本；保存草稿不影响数字员工，点击“发布生效”后才刷新引用该 Skill 的模板和实例。
- 移除模板与 Skill 的匹配关系后，该模板后续派生的实例不再拥有该模板技能；已有运行实例按运行时刷新策略更新。
- MVP 阶段不支持单个数字员工覆盖模板技能；后续如需例外能力，应通过“实例附加技能”单独建模。
- 创建数字员工时必须从模板出发，填写实例名称、目标团队、匹配项目和可选运行参数。

#### 5.3.5 展示

- 员工卡：展示 “N 项技能” 摘要。
- 员工详情面板：展示完整技能列表，标记来源（通用/模板匹配）。
- 员工页顶部统计条：员工总数、在线数、技能总数。
- 模板页：展示每个数字员工模板的职责、实例数、在线数、继承技能数和创建入口。
- 技能页：展示 Skill 配置工作台、技能资产列表和岗位技能匹配。

#### 5.3.6 管理

岗位管理入口位于左侧“软件工厂配置”区，菜单显示为“岗位”，页面标题为“数字员工模板（岗位）”，包含：

- **模板列表/卡片**：展示模板名称、模板类型、职责摘要、实例数量、在线数量、继承技能数量和启用状态。
- **创建数字员工**：基于模板创建实例，填写实例名称、目标团队、匹配项目和运行参数。
- **模板详情**：展示职责说明、默认运行参数、创建策略、模板技能摘要和影响实例。
- **跳转技能页**：带入当前模板过滤，查看“通用技能 + 模板匹配技能”的最终生效技能集。

技能管理入口为左侧配置区菜单“技能”，包含：

- **Skill 配置工作台**：参考配置页结构，左侧为项目文件树（`SKILL.md`、`scripts/`、`resources/`），右侧顶部维护 Skill 名称和描述，中间展示 YAML front matter 预览，下方使用 Markdown 编辑器维护 `SKILL.md` 正文。
- **技能资产列表**：表格展示全部 Skill（名称、描述、作用范围、引用模板数、引用实例数、版本、状态、更新时间），支持搜索/筛选。
- **岗位技能匹配**：选择数字员工模板后查看/编辑该模板已匹配的 Skill，支持关联已有 Skill、移除匹配、查看通用技能叠加结果。
- **发布生效**：Skill 正文保存为草稿后不影响运行实例；点击“发布生效”后生成新版本，并提示“将影响 N 个数字员工模板、M 个数字员工实例”。

### 5.4 核心业务对象

| 对象 | 核心字段/内容 | 说明 |
|---|---|---|
| Team | id、name、leader、members、healthy、currentProject、activities | 团队是协作和项目承接单元。 |
| Worker/Member | id、name、agentTemplateId、projectRole、teamId、projectId、status、currentTaskSummary、runtimeSession、skills[] | 数字员工实例；由数字员工模板创建，继承模板技能，可分配到不同团队和项目。 |
| Project | id、name、stage、health、codeRepo、modelRepo、docs、blockers | 项目挂载在团队下，承载开发上下文。 |
| Doc | id、category、title、status、author、reviewers、updatedTs、version | 文档是项目驱动开发的核心载体。 |
| Decision | id、teamId、requester、type、title、urgent、expiresAt、status、context、options | 决策项需要负责人确认后推动执行。 |
| Activity | time、desc | 团队动态与执行上下文的事件摘要。 |
| AgentTemplate | id、code、displayName、description、responsibilities[]、defaultRuntime、createPolicy、instanceCount、enabled | 数字员工模板，定义职责、默认能力、默认运行参数和实例化规则。 |
| Skill | id、name、description、version、status、scope、frontmatter、markdownContent、files[]、publishedVersion、createdAt、updatedAt | Skill 是独立 Markdown 能力资产；核心内容为 `SKILL.md`，可附带 scripts/resources。 |
| TemplateSkillMapping | agentTemplateId、skillId、source、enabled、effectiveVersion | 岗位模板-Skill 多对多关联，是模板技能的唯一主数据。Worker.skills[] 运行时派生 = 通用技能 + 该模板匹配技能。 |

## 6. 全量功能规格

### 6.1 总览页

#### 6.1.1 全局统计卡

**功能描述**：展示活跃团队、在线员工、任务吞吐、待决策等全局指标。

**交互行为**：

- 点击活跃团队跳转团队页。
- 点击在线员工跳转员工页。
- 点击待决策跳转待决策页。
- 任务吞吐在当前原型中为静态展示。

**验收条件**：

- 用户进入总览页后，应能看到 4 类核心指标。
- 可跳转指标点击后应进入对应页面。
- 静态指标应在后续实现计划中标记数据来源待补。

#### 6.1.2 协作全景拓扑

**功能描述**：以团队卡、Leader 节点、成员节点和待分配池展示所有团队协作关系。

**交互行为**：

- 点击 Leader 节点打开 Leader 详情抽屉。
- 点击成员节点打开员工详情抽屉。
- 点击 Leader 对话入口打开小云，并切换到对应 Leader target。
- hover 某团队时，其他团队视觉弱化。

**验收条件**：

- 5 个研发组均应在总览拓扑中可见。
- 每组应展示 Leader、成员、当前项目和团队状态。
- 待分配员工应独立呈现，不能混入已分配团队。

#### 6.1.3 团队动态流

**功能描述**：展示团队活动事件，并支持按团队过滤。

**交互行为**：

- 支持“全部/单团队/未分配”等过滤。
- 点击拓扑团队可联动过滤动态。
- 点击空白或全部恢复全局动态。

**验收条件**：

- 过滤后动态列表只展示对应范围事件。
- 用户能从动态看出最近发生的项目、文档或决策动作。

### 6.2 团队页

#### 6.2.1 团队卡列表

**功能描述**：以卡片展示每个团队的健康状态、Leader 状态、在线人数、当前任务、当前项目、文档完成度和仓库状态。

**交互行为**：

- 点击团队卡进入团队详情。
- 点击监控/详情类入口进入对应团队详情。
- “更多”操作当前为占位。

**验收条件**：

- 团队页应展示全部 5 个研发组。
- 每张团队卡至少包含团队名、Leader、健康状态、当前项目摘要。
- 用户可从团队卡进入对应团队详情。

#### 6.2.2 团队更多操作

**功能描述**：预留历史、导出、暂停、重启、设置等团队管理动作。

**当前状态**：原型中为 alert 占位。

**验收条件**：

- 需求阶段保留入口。
- 具体动作定义进入后续迭代或设计阶段补齐。

### 6.3 项目页

#### 6.3.1 项目索引表

**功能描述**：从各团队当前项目聚合生成项目列表，展示项目名、代码、文档、承接团队、状态、Leader、最近更新。

**交互行为**：

- 点击项目名进入承接团队详情。
- 点击承接团队进入团队详情。
- 点击代码仓跳转外部代码地址。
- 点击文档进入团队详情并切换到项目文档库 tab。

**验收条件**：

- 项目页应与团队当前项目保持一致，不出现孤立项目。
- 用户可从项目页直达项目文档库。
- 代码仓和文档入口应在视觉上可识别为可点击。

### 6.4 员工页

#### 6.4.1 全员检索与筛选

**功能描述**：展示全部数字员工，包括已加入团队成员与未分配员工。

**交互行为**：

- 支持关键词搜索。
- 支持按角色筛选。
- 支持按状态筛选。
- 支持按团队或加入状态分组。

**验收条件**：

- 搜索和筛选结果应实时更新。
- 已入组与未入组员工应可区分。
- 角色筛选中不应保留已废弃的 librarian 选项，除非作为兼容项明确标记。

#### 6.4.2 员工卡

**功能描述**：员工卡展示头像、姓名、项目职责、所属团队、当前状态、当前关注点和操作入口。

**交互行为**：

- 普通成员不提供直接对话能力；若保留“对话”入口，应表达为“通过 Leader 联系”或跳转到所属团队 Leader 小云会话。
- “监控”入口用于查看员工状态、当前关注点、最近活动和所属项目职责，不应承担直接指挥成员的能力。
- “更多”操作当前为占位，用于后续承载查看档案、调整职责、移出团队等管理动作。
- 未入组员工支持加入团队。

**验收条件**：

- 每个员工应具备可识别的人物感信息：头像、姓名、项目职责、状态、关注点。
- 普通员工卡不能出现会让用户误解为“直接与成员对话”的完成态入口。
- 若用户从员工卡发起沟通，系统必须明确该沟通由 Leader/主智能体承接。
- 占位操作不能误导为已完成能力，后续计划需单独拆解。

#### 6.4.3 加入团队

**功能描述**：将未分配员工加入指定团队。

**交互行为**：

- 用户点击加入团队。
- 系统展示团队选择。
- 确认后员工从未分配池转入对应团队成员列表。

**验收条件**：

- 加入后员工所属团队状态更新。
- 总览拓扑、团队详情和员工页应保持一致。

### 6.5 待决策页

#### 6.5.1 决策列表与过滤

**功能描述**：汇总所有需要负责人处理的决策请求。

**交互行为**：

- 支持按紧急度过滤。
- 支持按团队过滤。
- 支持按状态过滤。
- 点击列表项查看右侧详情。

**验收条件**：

- 待处理决策应在列表中明确突出。
- 过滤条件组合后结果正确。
- 紧急/过期项应有明显视觉提示。

#### 6.5.2 决策详情

**功能描述**：展示决策发起方、上下文、倒计时/过期状态、可选方案和操作按钮。

**交互行为**：

- 点击决策行后右侧详情刷新。
- 支持复制上下文。
- 用户选择候选方案并确认后，决策状态从 `pending` 流转为 `resolved`；若用户无法选择，应可标记为 `blocked` 或保留待处理。
- 超过处理期限且未处理的决策进入 `expired` 状态，并保留上下文供后续追溯。
- 决策结果应回写到相关团队动态，并在关联项目/文档/小云上下文中可追踪。

**验收条件**：

- 用户能理解该决策为什么需要处理、由谁提出、影响哪个团队/项目。
- 每个决策至少有一个可选处理方案。
- 用户处理决策后，应看到明确的状态反馈。
- 已处理决策不应继续以待处理提醒打扰用户。
- 决策处理记录应能在团队动态或关联上下文中被追踪。

#### 6.5.3 全局待决策提醒

**功能描述**：在侧边栏和底部提醒条展示待决策数量或紧急提示。

**交互行为**：

- 点击提醒进入待决策页。

**验收条件**：

- 待决策数量变化时，提醒入口应同步更新。
- 用户从任何页面都能发现紧急决策。

### 6.6 岗位页 / 数字员工模板页

**功能描述**：管理数字员工模板（AgentTemplate），展示模板职责、实例数量、在线状态、继承技能摘要、创建策略和是否启用。模板页回答“这个数字员工模板是什么、能创建什么数字员工、默认具备哪些能力、可以匹配哪些项目”。

**交互行为**：

- 左侧菜单显示为“岗位”，页面标题显示为“数字员工模板（岗位）”。
- 支持查看 explorer / fixer / oracle / designer 四类有效数字员工模板。
- 支持查看模板详情，包括职责说明、实例统计、关联技能、默认运行参数和影响实例。
- 支持从模板创建数字员工实例：填写实例名称、选择目标团队、选择匹配项目、确认默认技能集。
- 数字员工实例创建后继承模板技能；后续可分配到不同项目，形成“同模板、多实例、多项目”的工作方式。
- MVP 阶段模板新增/删除可作为占位，但职责编辑、启停、创建数字员工和技能视图应在页面上有清晰表达。
- 点击“查看技能”可跳转技能页并带入对应模板过滤。

**验收条件**：

- 岗位页允许使用“岗位”作为一级菜单名称，但页面标题必须明确“数字员工模板（岗位）”，避免与项目职责混淆。
- 模板页不出现 librarian 作为可选有效模板。
- 用户能从模板页理解：数字员工模板 → 创建数字员工实例 → 继承技能 → 匹配项目。
- 基于模板创建数字员工后，员工页、团队详情、总览拓扑中的实例数量和归属信息同步刷新。
- 模板页不直接编辑 Worker 运行中状态；运行状态由员工实例和运行时会话维护。

### 6.7 技能页

**功能描述**：管理 Skill 资产和岗位技能匹配。Skill 是 Markdown 能力包，核心文件为 `SKILL.md`，包含 YAML front matter 和正文说明，可附带 scripts/resources。技能页承载 Skill 配置、草稿保存、发布生效、资产列表、模板匹配和影响范围提示。

**交互行为**：

- Skill 配置 tab：左侧展示项目文件树（`SKILL.md`、`scripts/`、`resources`）和能力引用；右侧顶部编辑 Skill 名称和描述；中间展示 YAML 预览；下方使用 Markdown 编辑器编辑 `SKILL.md` 正文。
- 技能资产 tab：展示技能名称、描述、作用范围、引用数字员工模板、引用数字员工实例、版本、状态、更新时间。
- 模板匹配 tab：选择数字员工模板后展示“通用技能 + 模板技能”的最终技能集。
- 新建/编辑 Skill 时，默认进入 Skill 配置工作台，而不是简单表单抽屉。
- 保存草稿不影响 Agent；发布生效后生成新版本，并提示影响数字员工模板和数字员工实例数量。
- 停用 Skill 时必须提示影响模板和实例数；已停用 Skill 不进入 Worker.skills[] 派生结果。

**验收条件**：

- 技能页和模板页的数据口径一致，TemplateSkillMapping 是岗位技能匹配唯一主数据。
- Skill 编辑器必须能看到并编辑 Skill 名称、描述和 Markdown 正文。
- 发布 Skill 后，员工卡技能摘要、员工详情技能列表、模板页技能摘要同步刷新。
- 技能管理不放在设置页中。

### 6.8 设置页

**功能描述**：只读展示系统运行信息，包括版本、端口、连接数、持久化路径、checkpoint 等。

**交互行为**：无编辑交互。

**验收条件**：

- 设置页应明确为只读展示。
- 不应提供技能、模板或运行配置编辑入口。

### 6.9 团队详情：团队工作台

#### 6.7.1 团队 Header

**功能描述**：展示团队名、当前项目、项目阶段、健康状态、阻塞、仓库入口、文档协作摘要和待处理事项。

**交互行为**：

- 点击文档摘要切换到项目文档库。
- 点击待处理文档进入对应文档详情。

**验收条件**：

- Header 应让用户一眼判断团队当前项目是否健康。
- 文档和待处理事项应能作为进入工作内容的入口。

#### 6.7.2 左侧编队

**功能描述**：展示 Leader 与团队成员卡片，突出姓名、项目职责、状态、当前关注点和人物感。

**交互行为**：

- 点击成员切换右侧详情。
- 点击 Leader 可回到团队/Leader 视角。
- Leader 卡提供发起会话入口。

**验收条件**：

- 当前选中成员应有明确高亮。
- 左侧编队不承担复杂操作，只负责选择和状态速览。

#### 6.7.3 右侧详情

**功能描述**：展示当前选中智能体或团队视角的任务、状态、在读文档、最近产出、风险和执行上下文。

**交互行为**：

- 随左侧选中对象变化。
- 展示相关活动时间线。

**验收条件**：

- 默认进入团队详情时应选中 Leader 或团队视角。
- 切换成员后，右侧内容必须同步变化。

### 6.10 团队详情：项目文档库

#### 6.8.1 文档树

**功能描述**：按 specs、plans、decisions、reports、notes 分类展示项目文档。

**交互行为**：

- 支持分类折叠/展开。
- 点击文档后右侧详情切换。
- 文档项显示状态、版本和协作提示。

**验收条件**：

- 五类文档均有明确分类。
- 当前选中文档应高亮。
- 文档状态图标/标签应与状态枚举一致。

#### 6.8.2 文档预览与编辑

**功能描述**：支持 Markdown 预览和编辑模式切换。

**交互行为**：

- 用户可在预览/编辑之间切换。
- 编辑内容可实时或切换后反映到预览。

**验收条件**：

- 预览模式适合阅读。
- 编辑模式适合修改文档正文。
- 未保存修改应有清晰提示或保存入口。

#### 6.8.3 保存草稿与发布

**功能描述**：支持保存草稿、发布给 Agent、撤销发布和回滚入口。

**交互行为**：

- 保存草稿后给出反馈。
- 发布表示将当前文档版本作为 Agent 可读取、可执行或可参考的正式上下文。
- 发布后应生成团队动态，并更新文档协作/状态信息。
- 发布后短时间内可撤销；撤销后该版本不再作为新的 Agent 执行上下文，但历史记录应保留。
- 回滚当前为占位。

**验收条件**：

- 保存和发布必须是两个不同动作。
- 发布行为应表达“将文档作为 Agent 执行上下文”。
- 发布后用户应能看到该动作已经影响团队上下文，而不只是本地保存成功。
- 撤销发布后，用户应能理解撤销的是“本次发布生效”，不是删除文档历史。
- 回滚若未实现，应明确标记为待补全。

#### 6.8.4 文档协作与历史

**功能描述**：展示当前阅读者、最近编辑者、冲突提示，并提供历史抽屉查看文档变更因果链。

**交互行为**：

- 点击历史入口打开/关闭历史抽屉。
- 历史项按决策、阻塞等类型区分展示。

**验收条件**：

- 用户能看到文档协作状态。
- 历史信息应帮助理解文档为何变化。

### 6.9 小云对话

#### 6.9.1 窗口与入口

**功能描述**：小云是网站的全局智能体对话入口，支持浮动按钮、嵌入窗口和独立弹窗模式。

**交互行为**：

- 点击浮动按钮打开/关闭对话面板。
- 面板可拖动、缩放、最大化/还原。
- 支持弹窗模式，并共享同一 target/session。

**验收条件**：

- 用户可从任意页面打开小云。
- 弹窗和嵌入模式不能创建语义上不同的会话。

#### 6.9.2 会话目标

**功能描述**：支持全局助手和各团队 Leader 作为对话 target。

**交互行为**：

- 从团队/Leader 入口打开时自动切换 target。
- 切换 target 后加载对应配置和历史。

**验收条件**：

- 用户应明确知道当前在和谁对话。
- 不同 Leader 的会话上下文应区分。

#### 6.9.3 消息输入与发送

**功能描述**：支持文本输入、快捷消息、发送中止和深度思考视觉开关。

**交互行为**：

- Enter 发送，Shift+Enter 换行。
- 发送过程中按钮变为终止。
- 深度思考当前作为 UI toggle。

**验收条件**：

- 消息发送状态应可见。
- 用户应能中断正在进行的发送。

#### 6.9.4 图片附件

**功能描述**：支持 Ctrl+V 粘贴图片、文件选择、多图缩略图预览和删除。

**交互行为**：

- 最多保留 4 张图片。
- 图片随文本一起发送。

**验收条件**：

- 图片发送前可预览。
- 用户可删除误选图片。

#### 6.9.5 富文本、问题与决策卡片

**功能描述**：支持消息富文本渲染、状态 callout、问题卡、决策卡、多问题 wizard 和 interaction questions。

**交互行为**：

- 决策卡选项可自动生成回答协议。
- 多问题支持分题、review、提交。
- 必填题应校验。
- Esc 可取消当前 interaction 或关闭抽屉。

**验收条件**：

- AI 输出中的问题/决策不应只以纯文本呈现，应转化为可操作卡片。
- 用户提交前应能检查回答。

#### 6.9.6 连接与运行配置展示

**功能描述**：展示和配置小云连接模式、传输模式、交互模式、模型、token、sessionId、端口和开发者命令。

**交互行为**：

- 支持 bridge、mock、opencode 模式。
- 支持测试连接。
- 支持重新绑定/新建/重建 Leader session。
- 支持复制 TUI attach/curl 命令。

**验收条件**：

- 用户能判断当前连接状态。
- 运行配置属于当前 Team/Leader，不应误解为全局唯一配置。

## 7. 状态与枚举

| 类型 | 枚举 |
|---|---|
| 员工状态 | busy、idle、unclaimed、offline、online |
| 团队运行/健康状态 | online、offline、healthy、degraded |
| 项目阶段 | discovery、design、build、test、release、archived |
| 项目健康 | healthy、warning、critical、degraded、unknown |
| 文档分类 | specs、plans、decisions、reports、notes |
| 文档状态 | draft、in_review、approved、in_execution、done、archived、blocked、rejected |
| 阻塞严重度 | high、medium、low |
| 决策状态 | pending、expired、resolved、blocked |
| 决策类型 | 需求歧义、偏好选择、不可逆操作 |
| 决策选项样式 | primary、normal、danger |
| 文档协作态 | stable、active、waiting、conflict |
| 小云模式 | bridge、mock、opencode |
| 小云传输模式 | normalized、passthrough、direct-opencode |
| 小云交互模式 | text-protocol、native-question |

## 8. 已知占位与待补全清单

| 模块 | 占位/问题 | 后续处理 |
|---|---|---|
| 团队更多操作 | 历史、导出、暂停、重启、设置等为 alert 占位 | 设计阶段定义动作边界和权限。 |
| 员工卡操作 | 对话、监控、更多为 alert 占位 | 明确普通成员是否允许直接对话；当前原则为不直接对话。 |
| 移除员工 | confirmRemoveWorker 仅确认，未实际移除 | 补充移除后的团队/项目影响规则。 |
| 文档回滚 | rollbackDoc 为 alert 占位 | 设计文档版本机制后再实现。 |
| 决策提交 | 决策详情候选按钮偏展示 | 补充状态流转和提交后反馈。 |
| 决策状态 Overlay | 有视觉层但缺少触发链路 | 判断是否保留，若保留需接入真实决策状态。 |
| 任务吞吐统计 | 当前为静态 mock | 后续设计统计口径。 |
| librarian 残留 | 样式/筛选可能仍存在 | 清理 UI 残留或标记兼容。 |
| Leader 抽屉预览 | 可能读取错误数据源 | 实现前修正数据来源。 |

## 9. 全局验收清单

1. 用户能通过总览理解 5 个研发组、员工池、当前项目和待决策状态。
2. 用户能从团队页进入任一团队详情，并查看团队工作台和项目文档库。
3. 用户能从项目页查看所有团队当前项目，并直达项目文档库。
4. 用户能在员工页搜索、筛选、分组查看全部数字员工。
5. 用户能将未分配员工加入指定团队，并在相关页面看到一致状态。
6. 用户能在待决策页查看、过滤和理解所有决策请求。
7. 用户能在项目文档库中查看、编辑、保存草稿、发布文档，并查看协作状态与历史。
8. 用户能从全局或 Leader 入口打开小云，并明确当前对话 target。
9. 小云应支持文本、图片、富文本、问题卡、决策卡和连接状态展示。
10. 设置页只读展示系统运行信息，不承担配置编辑。
11. 所有原型中仅占位的操作必须在实现计划中单独标记，不能默认视为已完成。
12. 需求规格不包含技术实现细节；技术架构与 agent-web-kit 集成策略进入后续设计文档。
