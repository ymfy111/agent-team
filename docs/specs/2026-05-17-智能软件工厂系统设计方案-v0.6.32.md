# 智能软件工厂系统设计方案

> 版本：v0.6.32  
> 日期：2026-05-17  
> 状态：系统设计正本 / AI 原生岗位与任务单闭环版

---

## 1. 设计目标

本设计方案定义智能软件工厂的核心业务对象、状态流、运行时边界、任务单机制和质量门禁机制。v0.6.32 的重点是把当前半手工多智能体协作方式沉淀为未来系统可自动调度的模型。

核心目标：

```text
1. 用 AI 原生岗位替代人类岗位照搬。
2. 将 Leader、岗位、数字员工、任务单、进度、阻塞、决策、验收和活动回写建模。
3. 支持 OpenCode 独立进程作为 Worker 的主智能体运行体。
4. 支持 OpenCode 内部子智能体作为运行时内部能力，而不是组织级 Worker。
5. 支持 agent-browser / BrowserRuntime 作为开发自测和交付审查能力。
```

---

## 2. 总体闭环

```text
Skill
→ AgentTemplate / AI 原生岗位模板
→ Worker / 数字员工实例
→ WorkerRuntimeBinding / OpenCode 主智能体运行体
→ Team / Project
→ Doc 发布
→ WorkOrder 派发
→ Progress / Blocker / Decision / ChangeRequest
→ AcceptanceRecord / ReworkOrder
→ Activity 回写
```

## 3. AI 原生岗位模型

### 3.1 AgentTemplate / 岗位模板

```ts
type AgentTemplate = {
  id: string
  code: 'collaboration_planner' | 'implementation_validator' | 'delivery_reviewer'
  name: '协同规划岗' | '实现验证岗' | '交付审查岗'
  description: string
  defaultWorkModes: WorkMode[]
  skillIds: string[]
  enabled: boolean
}
```

岗位说明：

| code | 中文名 | 说明 |
|---|---|---|
| collaboration_planner | 协同规划岗 | 默认担任 Leader，负责沟通、设计、计划、任务、进度、阻塞、分歧初判。 |
| implementation_validator | 实现验证岗 | 负责实现、TDD、自测、修复、执行回执，可多实例并行。 |
| delivery_reviewer | 交付审查岗 | 负责独立验收、质量门禁、整改建议、初验收报工。 |

### 3.2 TeamRole / 团队角色

```ts
type TeamRole = 'leader' | 'member'
```

规则：

```text
Leader / 组长不是单独岗位，而是团队角色。
一个团队必须有且只有一个 Leader。
Leader 默认由协同规划岗 Worker 担任。
```

## 4. Worker 与 OpenCode Runtime

### 4.1 Worker

```ts
type Worker = {
  id: string
  name: string
  templateId: string
  teamId?: string
  teamRole: 'leader' | 'member'
  lifecycleStatus: 'draft' | 'enabled' | 'disabled'
  runtimeStatus: 'offline' | 'starting' | 'idle' | 'busy' | 'blocked' | 'failed'
  currentWorkOrderId?: string
  currentTaskSummary?: string
}
```

### 4.2 WorkerRuntimeBinding

```ts
type WorkerRuntimeBinding = {
  id: string
  workerId: string
  runtimeType: 'opencode' | 'mock' | 'custom'
  runtimeHostId: string
  workspaceDir: string
  primaryModelProfileId?: string
  allowedSubAgentModelProfileIds?: string[]
  skillSnapshotId: string
  syncStatus: 'synced' | 'pending' | 'failed'
  lastSyncAt?: string
}
```

一个 Worker 对应一个具体 OpenCode 独立进程 / workspace / 主智能体运行体。OpenCode 内部可根据规则调用子智能体，但子智能体是运行时内部能力，不默认上升为智能软件工厂的正式 Worker。

## 5. WorkMode / 工作模式

岗位是稳定身份，工作模式是任务中的专业视角。

```ts
type WorkMode =
  | 'requirements_clarification'
  | 'system_design'
  | 'planning'
  | 'architecture_review'
  | 'dev_test'
  | 'code_review'
  | 'browser_self_check'
  | 'delivery_acceptance'
  | 'rework_planning'
```

示例：

```text
协同规划岗：requirements_clarification / system_design / planning / architecture_review
实现验证岗：dev_test / code_review / browser_self_check
交付审查岗：delivery_acceptance / rework_planning
```

## 6. WorkOrder / 任务单模型

```ts
type WorkOrder = {
  id: string
  title: string
  projectId: string
  teamId: string
  phaseId: string

  sourceType: 'plan' | 'decision' | 'change_request' | 'acceptance_fix' | 'bugfix' | 'tech_debt'
  sourceRefId?: string

  coordinatorWorkerId: string        // 通常是 Leader / 协同规划岗
  primaryTemplateId: string          // 主责岗位
  assigneeWorkerIds: string[]        // 具体执行数字员工
  supportingTemplateIds?: string[]

  requiredWorkModes: WorkMode[]
  reviewStrategy: 'self_check' | 'subagent_review' | 'independent_worker_review'
  acceptanceStrategy: 'none' | 'independent_acceptance'

  priority: 'P0' | 'P1' | 'P2'
  dependsOnWorkOrderIds: string[]
  canRunInParallelWith?: string[]

  objective: string
  inputDocIds: string[]
  scope: string[]
  outOfScope: string[]
  expectedOutputs: string[]
  selfCheckRequirements: string[]
  acceptanceCriteria: string[]

  browserRequired?: boolean
  browserSessionPolicy?: 'reuse_worker_session' | 'fresh_session' | 'independent_acceptance_session'
  evidenceRequired?: boolean

  status: 'draft' | 'assigned' | 'running' | 'blocked' | 'submitted' | 'reviewing' | 'accepted' | 'rework_required' | 'done' | 'cancelled'
  createdAt: string
  updatedAt: string
}
```

设计规则：

```text
1. WorkOrder 是计划进入执行阶段后的最小调度单元。
2. WorkOrder 先挂主责岗位，再绑定具体 Worker。
3. 实现验证岗可自测，但不能自己完成交付放行。
4. 交付审查采用批量门禁，不做碎片化即时往返。
5. 验收不通过生成 ReworkOrder 或新的 acceptance_fix 类型 WorkOrder。
```

## 7. 进度、阻塞和分歧

### 7.1 WorkOrderProgress

```ts
type WorkOrderProgress = {
  workOrderId: string
  status: 'assigned' | 'running' | 'blocked' | 'submitted' | 'reviewing' | 'accepted' | 'rework_required' | 'done'
  percent?: number
  currentStep?: string
  blockerIds?: string[]
  lastActivityId?: string
  updatedAt: string
}
```

### 7.2 Blocker

```ts
type Blocker = {
  id: string
  workOrderId: string
  raisedByWorkerId: string
  title: string
  description: string
  severity: 'low' | 'medium' | 'high'
  status: 'open' | 'leader_reviewing' | 'resolved' | 'converted_to_decision'
  createdAt: string
}
```

处理规则：

```text
普通执行问题：实现验证岗自处理。
任务范围、需求不清、实现路径分歧：提交 Leader 初判。
超出 Leader 判断范围：生成 Decision / ChangeRequest，上报用户。
```

## 8. Decision 与 ChangeRequest

### 8.1 Decision

Decision 用于人在环确认：

```text
需求取舍
优先级调整
不可逆操作
实现路径重大分歧
交付范围变化
```

### 8.2 ChangeRequest

```ts
type ChangeRequest = {
  id: string
  projectId: string
  sourceType: 'user_feedback' | 'implementation_blocker' | 'acceptance_issue' | 'architecture_risk'
  sourceRefId?: string
  title: string
  description: string
  impactDocIds: string[]
  impactWorkOrderIds: string[]
  status: 'draft' | 'assessing' | 'approved' | 'rejected' | 'applied'
  createdByWorkerId: string
  createdAt: string
}
```

变更闭环：

```text
发现变化
→ 生成 ChangeRequest
→ Leader 评估影响
→ 必要时上报 Decision
→ 更新文档 / 计划 / 任务单
→ 生成新 WorkOrder 或整改任务单
```

## 9. AcceptanceRecord / 交付审查

```ts
type AcceptanceRecord = {
  id: string
  workOrderId: string
  reviewerWorkerId: string
  status: 'passed' | 'rework_required' | 'blocked'
  summary: string
  findings: Array<{
    severity: 'low' | 'medium' | 'high'
    title: string
    detail: string
    recommendation: string
  }>
  evidenceRefs: string[]
  reworkWorkOrderIds?: string[]
  createdAt: string
}
```

规则：

```text
1. 交付审查岗不参与实现验证岗的日常 TDD 循环。
2. 交付审查使用任务单验收标准批量检查。
3. 不通过时输出完整问题清单和整改建议。
4. 通过时记录初验收结果，允许进入下一阶段或用户确认。
```

## 10. BrowserRuntime / 浏览器运行时

浏览器能力分两层：

```text
BrowserRuntime
  底层浏览器能力，P0a/P0b 默认以 agent-browser 作为 provider。

Browser QA / Acceptance Skill
  实现验证岗和交付审查岗如何使用浏览器的工作规程。
```

```ts
type BrowserRuntimeSession = {
  id: string
  ownerWorkerId: string
  workspaceId: string
  provider: 'agent-browser' | 'gstack-browser' | 'playwright' | 'browserbase' | 'custom'
  endpointRef?: string
  status: 'starting' | 'ready' | 'busy' | 'idle' | 'error' | 'stopped'
  profileDir?: string
  lastUsedAt: string
}
```

规则：

```text
实现验证岗可以复用自己的浏览器 session 做自测。
交付审查岗应使用 independent_acceptance_session，避免被开发过程状态污染。
并行任务按 worker / workspace 隔离浏览器实例。
```

