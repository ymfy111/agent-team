# ADR-0007：岗位、计划、任务单与交付验收模型

> 版本：v0.6.32  
> 日期：2026-05-17  
> 状态：Accepted

---

## 背景

当前 `agent-team / 智能软件工厂` 项目本身正在以半手工方式进行多智能体协作：项目负责人手动调度设计智能体、编程智能体和评审智能体完成需求、设计、编码、测试和验收。

这暴露出智能软件工厂未来必须解决的问题：智能体不能只靠一个通用提示词工作，而需要明确岗位、项目上下文、阶段计划、任务单、执行回执和验收门禁。

---

## 决策

采用以下协作模型：

```text
岗位 / 角色
→ 工作指南
→ 项目上下文
→ 阶段计划
→ 任务单 WorkOrder
→ 执行回执
→ 交付验收
→ 活动回写 / 整改任务单
```

核心岗位收敛为：

```text
需求设计岗：需求 + 设计合并，负责 PRD、系统设计、施工图、计划和任务单。
开发测试岗：编码 + 测试合并，负责 TDD、实现、自测和执行回执，可多实例并行。
交付验收岗：独立质量门禁，负责整改意见、整改任务单和初验收报工。
```

开发计划先挂岗位，具体任务单再绑定数字员工实例。

---

## 约束

```text
1. 工作指南不是任务单。
2. 开发测试岗可以自测，但不能自我放行。
3. 交付验收岗不参与开发测试岗的日常 TDD 循环。
4. 验收不通过时输出整改意见和整改任务单。
5. 验收通过时记录初验收报工并允许进入下一阶段。
6. P0a 可先用 Markdown 任务单模拟 WorkOrder，P0b/P1 再持久化。
```

---

## 影响

- `AgentTemplate` 需要支持 `requirement_design`、`dev_test`、`delivery_acceptance` 等 roleKey。
- `WorkOrder` 需要支持主责岗位、协作岗位、验收岗位和执行回执。
- `Activity` 需要支持任务派发、任务完成、验收失败、验收通过、整改请求等事件。
- P0a 任务文档必须按多智能体协作方式写清楚岗位责任。
