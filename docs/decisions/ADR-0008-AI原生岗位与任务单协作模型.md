# ADR-0008：AI 原生岗位与任务单协作模型

> 状态：Accepted  
> 日期：2026-05-17  
> 版本：v0.6.32

---

## 1. 背景

早期设计中，智能软件工厂曾沿用人类软件工程岗位：Leader、架构、开发、测试、设计、建模等。但在大模型场景下，过细的人类岗位映射会导致沟通交接成本上升，甚至超过任务本身成本。

大模型智能体具备跨职能能力，更适合以少量复合岗位完成端到端工作，并在关键节点通过独立审查、独立验收和质量门禁控制风险。

同时，当前 `agent-team` 项目本身是智能软件工厂的半手工 POC：项目负责人手动调度设计智能体、OpenCode / Codex、评审/验收视角协作。该过程应沉淀为未来系统的产品模型。

---

## 2. 决策

智能软件工厂采用 AI 原生岗位模型。

核心岗位为：

```text
协同规划岗
实现验证岗
交付审查岗
```

并定义：

```text
Leader / 组长 是团队角色，不是单独岗位。
Leader 默认由协同规划岗 Worker 担任。
```

任务协作以 WorkOrder / 任务单为最小调度单元。任务单连接：

```text
项目
阶段
岗位
具体 Worker
输入文档
执行范围
自测要求
验收标准
进度
阻塞
决策
交付审查
活动回写
```

---

## 3. 岗位说明

### 3.1 协同规划岗

负责用户沟通、需求沉淀、系统设计、阶段计划、任务单、进度跟踪、阻塞协调、分歧初判和对人反馈。

### 3.2 实现验证岗

负责编码、TDD、自测、局部集成、修复和执行回执。可多个实例并行。

### 3.3 交付审查岗

负责独立验收、质量门禁、整改建议和初验收报工。不参与实现验证岗的日常 TDD 循环。

---

## 4. 质量门禁原则

```text
1. 实现验证岗的自测不等于交付放行。
2. 交付审查岗采用批量门禁，不做碎片化即时往返。
3. 验收不通过时输出完整问题清单和整改建议。
4. 验收不通过时生成整改任务单，而不是让开发和验收无限即时循环。
5. 高保障任务可以使用独立 Worker、独立会话或不同模型执行交付审查。
```

---

## 5. OpenCode 与子智能体

一个 Worker 可以绑定一个 OpenCode 独立进程 / workspace / 主智能体。

OpenCode 内部可以按规则调用子智能体处理局部事务，例如代码审查、测试补充、风险扫描。但子智能体是运行时内部能力，不默认成为智能软件工厂的正式 Worker。

---

## 6. 浏览器验收

浏览器能力按两层设计：

```text
BrowserRuntime
  底层浏览器运行时。P0a/P0b 默认以 agent-browser 作为 provider。

Browser QA / Acceptance Skill
  规定实现验证岗和交付审查岗如何使用浏览器做自测和验收。
```

P0a 只保留模型字段和指南，不真实调用 agent-browser。

---

## 7. 影响

### 正向影响

```text
1. 岗位与数字员工映射更直观。
2. 岗位数量少，沟通成本低。
3. WorkOrder 可以明确任务、进度和验收标准。
4. 交付审查独立性更强。
5. Mock 数据和原型可以直观展示软件工厂的核心价值。
```

### 代价

```text
1. 传统人类岗位名需要从主模型中淡化。
2. 需要在文档和任务单中明确工作模式，避免一个复合岗位职责过大。
3. 需要通过交付审查和质量门禁弥补少岗位带来的独立性风险。
```

---

## 8. 状态

该决策从 v0.6.32 起生效。后续 P0a mock seed、contracts 类型、domain selector 和原型均应以此为准。
