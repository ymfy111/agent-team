# 基于 v0.6.32 的原型就地优化执行清单

> 用途：给后续执行者 / 编程智能体使用。  
> 基线：v0.6.32 最新稳定原型。  
> 执行方式：只在 v0.6.32 原型上就地优化，不新增并行原型，不升版本号。  
> 输出目标：完成一个可评审、可运行、语义收敛的 v0.6.32 原型更新版。

---

## 0. 执行前锁定条款

执行前必须先确认以下 5 条硬约束，不允许在执行过程中自行扩展或改口径。

### 0.1 协同规划岗只作为组长存在

- 协同规划岗 = 团队 Leader = 组长。
- 每个团队最多 1 个协同规划岗。
- 协同规划岗不作为普通成员出现。
- 协同规划岗不进入普通成员池。
- 不再存在“普通协同规划岗”。
- 系统内可以有多个协同规划岗实例，但每个团队只能绑定一个。

示例：

```text
研发一组：管理1 / 协同规划岗 / 组长
研发二组：管理2 / 协同规划岗 / 组长
研发三组：管理3 / 协同规划岗 / 组长
```

---

### 0.2 技术专家岗默认是共享专家池，按需介入

- 系统架构师 / 技术专家岗默认不是每个团队固定成员。
- 只有出现重大技术决策、架构分歧、关键方案评审时，才显示为“已介入”。
- 未介入时，团队卡可以显示为：

```text
专家支持：待介入 / 暂无需介入
```

- 不要把每个团队都强制做成固定 4 岗齐备。

---

### 0.3 待审查与待决策必须分开表达

- 待审查：交付审查岗的质量门禁队列。
- 待决策：需要组长、技术专家或用户作出选择或确认的问题。
- 交付审查不通过时，优先退回实现验证岗整改。
- 只有涉及业务取舍、重大风险、架构方向、用户偏好时，才升级为待决策。

典型流转：

```text
交付审查不通过 → 退回实现验证岗整改
实现验证岗无法判断 → 提交协同规划岗
协同规划岗无法判断 → 提交系统架构师 / 技术专家岗
涉及业务取舍 → 升级给用户决策
```

---

### 0.4 岗位色与状态色必须分离

岗位色只表达岗位类型，状态色只表达运行状态。

岗位色建议：

```text
协同规划岗：紫蓝
实现验证岗：蓝色
交付审查岗：橙色
系统架构师 / 技术专家岗：深蓝 / 靛色
```

状态色建议：

```text
在线：绿色
忙碌：紫 / 蓝紫
离线：灰色
阻塞：红色
```

头像光条只做数字员工特征和弱状态联动，不承担主要状态表达。

---

### 0.5 本轮只做 v0.6.32 原型就地优化

本轮不升版本，不新增页面，不真实接入后端、Gateway、OpenCode Runtime 或 agent-web-kit。

允许输出：

```text
docs/prototypes/Agent-Team-V2-注册中心-v0.6.32.html
```

如需避免覆盖，也可以先输出：

```text
docs/prototypes/Agent-Team-V2-注册中心-v0.6.32-adjusted.html
```

但页面内标题、版本徽标、文档说明口径必须保持：

```text
v0.6.32 最新稳定基线
```

严禁再出现 v0.6.33 作为当前正式原型版本。

## 1. 本轮目标

把 v0.6.32 原型收敛为一个能清楚表达以下理念的智能软件工厂原型：

```text
岗位驱动
数字员工实例化
多智能体协同
文档作为协作上下文
组长负责规划与协调
实现验证岗负责开发 + 自测
交付审查岗负责初验收与质量门禁
技术专家岗按需介入重大技术判断
小云是全局助手，不是团队成员
```

## 3. 全局岗位体系

统一收敛为 4 类 AI 原生岗位。

| 岗位 | 定位 | 是否可多实例 | 是否可做组长 | 执行规则 |
|---|---|---:|---:|---|
| 协同规划岗 | 用户沟通、需求设计、计划拆解、任务分派、进度跟踪、分歧协调 | 是 | 是，且只能作为组长 | 每个团队最多 1 个 |
| 实现验证岗 | 开发、TDD、自测、修复、提交实现结果 | 是 | 否 | 可以多个并行 |
| 交付审查岗 | 初验收、质量门禁、整改建议、通过后报工 | 是 | 否 | 不等同于开发自测 |
| 系统架构师 / 技术专家岗 | 重大技术决策、架构分歧、关键方案评审 | 是 | 否 | 默认共享专家池，按需介入 |

## 12. 验收标准

### 12.1 页面级验收

```text
总览页：
  能看出全局团队运行态
  能看出每个团队的组长与岗位构成
  动态体现智能体协同闭环

团队页：
  每个团队卡显示组长、项目、阶段、岗位构成、专家支持状态

团队详情：
  左侧编队无普通协同规划岗
  组长明确是协同规划岗
  小云没有进入团队成员
  右侧详情随岗位切换有不同内容

项目文档库：
  文档分类体现需求设计 / 计划任务 / 实现验证 / 交付审查 / 决策变更
  发布语义明确是给 Agent 的执行上下文

项目页：
  项目仍从团队派生
  不出现独立项目主数据心智

待决策页：
  决策体现岗位升级路径
  待审查与待决策语义分开

岗位页：
  清楚表达岗位模板、职责边界、是否可多实例、是否可做组长

员工页：
  岗位筛选收敛
  协同规划岗只作为组长出现
  普通员工沟通入口转为通过组长联系

技能页：
  能看出技能资产、适用岗位、引用实例数

设置页：
  仍保持只读运行信息

小云：
  只作为全局助手
  不混入团队成员
```

### 12.2 视觉验收

```text
头像风格符合已确认设计图
小云是机器人
组长是深色人形数字员工
普通员工是浅色人形数字员工
太阳穴光点统一
衣服光条克制
岗位色与状态色分离
状态由 UI 状态点 / 标签表达
```

```text
用户能理解：
  谁是组长
  谁负责实现
  谁负责交付审查
  谁负责技术把关
  技术专家不是默认每队固定成员
  小云不是团队成员
  文档发布才会影响 Agent 执行
  当前项目本身也是智能软件工厂的 POC
```
