# 智能软件工厂开发计划

> 版本：v0.6.32  
> 日期：2026-05-14  
> 依据：`2026-05-13-智能软件工厂产品需求规格-v0.6.32-revised.md`、`2026-05-13-智能软件工厂系统设计方案-v0.6.32-revised.md`、`Agent-Team-V2-注册中心-v0.6.32-ui-review-task-breakdown.html`

## 1. 开发目标

本轮开发目标是把原型从“功能演示”推进到“可拆解、可联调、可验收”的前端 MVP。重点不是一次性接入完整后端，而是先把运行态与配置态的产品边界、数据模型、页面交互和验收链路固化。

## 2. 版本范围

### 2.1 P0a：前端可演示 MVP

- 左侧导航分组：运行态 / 软件工厂配置。
- 运行态页面：总览、团队、项目、待决策。
- 配置态页面：岗位、员工、技能、设置。
- 岗位页：展示数字员工模板、职责、实例数、继承技能数、创建数字员工入口。
- 技能页：Skill 配置工作台、SKILL.md 编辑器、技能资产列表、岗位技能匹配。
- 员工页：展示数字员工实例、搜索筛选、技能摘要、团队/项目归属。
- 文档库：保留草稿、发布给 Agent、历史抽屉的前端交互。
- 小云：保留 mock / bridge 配置入口，不强依赖真实后端。

### 2.2 P0b：可运行 MVP

- 将 Team、Worker、AgentTemplate、Skill、TemplateSkillMapping、Doc、Decision、Activity 接入 API。
- Skill 草稿、发布版本、影响范围持久化。
- 基于岗位创建数字员工实例时写入 `agentTemplateId、teamId、projectId、runtimeSession`。
- 文档发布版本持久化，并可被小云 / Leader 会话读取。
- 决策处理结果回写团队动态、文档历史和小云上下文。

### 2.3 P1：增强版

- 岗位创建数字员工向导完整化。
- Skill scripts/resources 文件管理。
- 技能版本回滚、差异对比和发布审批。
- 员工实例附加技能或临时技能覆盖。
- 权限、审计、团队操作、模板启停和影响分析。

## 3. 前端开发拆解

| 模块 | 优先级 | 主要任务 | 验收要点 |
|---|---:|---|---|
| 导航与布局 | P0a | 左侧菜单分组；配置区视觉区分；tab 标题同步 | 岗位/员工/技能/设置位于下半部分工厂配置区 |
| 总览页 | P0a | 指标卡、协作全景、团队动态过滤 | 点击指标和拓扑可跳转对应页面 |
| 团队页 | P0a | 团队卡、团队详情入口、Header 状态 | 5 个团队可见，进入详情默认选中 Leader |
| 项目页 | P0a | 项目索引、代码/文档入口 | 文档入口进入团队详情 docs tab |
| 员工页 | P0a | 全量实例、搜索、筛选、技能摘要、加入团队 | 不再出现有效 librarian 筛选项；员工技能由模板派生 |
| 岗位页 | P0a | 数字员工模板卡、实例统计、创建入口、查看技能 | 用户能理解“岗位 → 创建数字员工 → 继承技能 → 匹配项目” |
| 技能页 | P0a | Skill 工作台、SKILL.md、资产列表、岗位技能匹配 | 保存草稿不影响实例；发布生效提示影响范围 |
| 待决策页 | P0a | 决策列表、详情、状态流转 | pending 可处理为 resolved/blocked/expired |
| 设置页 | P0a | 只读运行信息 | 不出现岗位/员工/技能编辑入口 |
| 小云入口 | P0a | 全局浮层、Leader target、配置面板 | 普通成员沟通必须转到 Leader |

## 4. 数据模型开发顺序

1. `AgentTemplate`：数字员工模板/岗位模板。
2. `Skill`：独立 Skill Markdown 能力资产。
3. `TemplateSkillMapping`：岗位模板与 Skill 的唯一匹配主数据。
4. `Worker`：数字员工实例，引用 `agentTemplateId`。
5. `ProjectAssignment`：实例与团队/项目的匹配关系。
6. `DocVersion`：草稿、发布版本和历史。
7. `Decision` / `Activity`：决策闭环和事件追踪。

## 6. 联调计划

### 6.1 Mock 阶段

- 使用 mock seed 初始化 teams、workers、agentTemplates、skills、mappings、decisions。
- 所有 UI 状态先完成本地联动。
- 完成截图评审：总览、员工、岗位、技能、团队详情、文档库、待决策。

### 6.2 API 阶段

- 先接只读 API：Team / Worker / AgentTemplate / Skill。
- 再接写 API：创建 Worker、保存 Skill 草稿、发布 Skill、处理 Decision。
- 最后接事件：Activity、文档发布、小云 question / decision 事件。

## 7. UI 评审与截图要求

每次迭代至少输出以下截图用于设计评审：

- 总览页：验证运行态入口与协作全景。
- 员工页：验证配置区中数字员工实例管理心智。
- 岗位页：验证“数字员工模板（岗位）”命名和创建链路。
- 技能页：验证 Skill 配置页是否接近 SKILL.md 工作台形态。
- 团队详情：验证运行态观察和配置区职责没有混淆。

截图评审要同时检查功能完整性和视觉层级，包括菜单分组、标题文案、主要按钮、影响提示、空状态和占位能力标记。

## 8. 测试计划

| 用例 | 操作 | 期望 |
|---|---|---|
| NAV-001 | 点击员工/岗位/技能/设置 | 均位于配置区，打开对应 tab |
| TEMPLATE-001 | 在岗位页点击创建数字员工 | 出现创建向导或明确占位提示 |
| TEMPLATE-002 | 在岗位页点击查看技能 | 跳转技能页并切到岗位技能匹配 |
| SKILL-001 | 编辑 SKILL.md 并保存草稿 | 草稿状态更新，但不刷新 Worker.skills[] |
| SKILL-002 | 发布生效 | 出现影响 N 个岗位模板、M 个实例的提示 |
| WORKER-001 | 员工筛选角色 | 不出现有效 librarian 选项 |
| DECISION-001 | 处理待决策 | 状态更新并回写团队动态 |
| DOC-001 | 发布文档给 Agent | published version 更新，历史抽屉新增记录 |
