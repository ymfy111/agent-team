# 智能软件工厂开发计划

> 版本：v0.6.32  
> 日期：2026-05-14  
> 状态：P0a 开工准备稿  
> 依据：PRD v0.6.32、系统设计 v0.6.32、原型 v0.6.32、技术落地补充评审意见  
> 关联任务拆解：`docs/tasks/2026-05-14-智能软件工厂开发任务拆解-v0.6.32.md`

## 1. 开发目标

本轮开发目标是把原型从“功能演示”推进到“可拆解、可联调、可验收”的前端 MVP。重点不是一次性接入完整后端，而是先把运行态与配置态的产品边界、数据模型、页面交互和验收链路固化。

P0a 的核心不是页面数量，而是验证智能软件工厂的最小业务闭环：

```text
Skill → AgentTemplate → Worker + RuntimeBinding → Team/Project → Doc 发布 → Decision 处理 → Activity 回写
```

## 2. 阶段范围

### 2.1 P0a：前端可演示 MVP

P0a 使用 mock seed、本地状态和 MockChatAdapter 跑通产品闭环。

- 左侧导航分组：运行态 / 软件工厂配置。
- 运行态页面：总览、团队、项目、待决策。
- 配置态页面：岗位、员工、技能、设置。
- 岗位页：展示数字员工模板、职责、实例数、继承技能数、创建数字员工入口。
- 技能页：Skill 配置工作台、SKILL.md 编辑器、技能资产列表、岗位技能匹配。
- 员工页：展示数字员工实例、搜索筛选、技能摘要、团队 / 项目归属、运行时工作目录与技能同步状态。
- 文档库：草稿、发布给 Agent、历史抽屉的前端模拟交互。
- 小云：使用 MockChatAdapter，验证入口、target 和 HostContext。

### 2.2 P0b：本地可运行 MVP

P0b 引入本地 API 和 SQLite 持久化，让核心业务数据可恢复。

- 引入 `apps/api`：Fastify 本地 API。
- 引入 `db/schema.prisma`：SQLite + Prisma。
- 持久化 AgentTemplate、Worker、RuntimeHost、WorkerRuntimeBinding、Skill、SkillVersion、TemplateSkillMapping、Team、Project、ProjectAssignment、Doc、DocVersion、Decision、Activity。
- Skill 草稿、发布版本、影响范围持久化。
- 文档发布版本持久化。
- 决策处理结果回写团队动态，并记录 `handledBy`、`handledAt`。
- 小云可继续使用 mock adapter；可选接入 agent-web-kit 文本对话，但不作为 P0b 阻塞项。
- RuntimeHost 可先支持单机；Worker 创建时可初始化本地 OpenCode workspaceDir 和 skills 快照。

### 2.3 P1：服务化与真实集成增强

- PostgreSQL + Prisma。
- 权限、审计、团队操作、模板启停和影响分析。
- agent-web-kit + agent-gateway + OpenCode 真实链路。
- 多 RuntimeHost / RuntimeNode 调度、容量标签、跨节点技能同步和失败重试。
- interaction → Decision → 回写小云 / 文档 / Activity 的完整闭环。
- Skill scripts/resources 文件管理、版本回滚、差异对比和发布审批。
- 员工实例附加技能或临时技能覆盖。

## 3. P0a 完成定义（Definition of Done）

P0a 完成不以“页面做得多完整”为标准，而以最小业务闭环是否可演示、可验收为标准。

| 编号 | 完成标准 |
|---|---|
| DoD-01 | 可以从 mock seed 初始化完整业务数据。 |
| DoD-02 | 可以浏览运行态页面：总览 / 团队 / 项目 / 待决策。 |
| DoD-03 | 可以浏览配置态页面：岗位 / 员工 / 技能 / 设置。 |
| DoD-04 | 可以创建或编辑一个 Skill 草稿，并模拟发布。 |
| DoD-05 | 可以将 Skill 挂载到 AgentTemplate，并通过 TemplateSkillMapping 派生技能。 |
| DoD-06 | 可以基于 AgentTemplate 创建 Worker。 |
| DoD-07 | 可以将 Worker 加入 Team，并匹配 Project。 |
| DoD-07a | 可以在 mock 数据中看到 Worker 的 RuntimeHost、workspaceDir、skillDir、技能初始化快照和同步状态。 |
| DoD-08 | 可以编辑 Doc 草稿并模拟发布。 |
| DoD-09 | 可以生成并处理一条 Decision。 |
| DoD-10 | 可以在 Activity 中看到关键业务动作回写。 |
| DoD-11 | 小云使用 MockChatAdapter，能验证入口、target 和 HostContext。 |
| DoD-12 | 不依赖真实后端、真实数据库、真实 Gateway、真实 OpenCode。 |

### 3.1 P0a 不做范围

P0a 明确不做以下内容：

```text
- 真实登录 / 权限 / 多用户。
- 真实后端 API。
- 真实数据库。
- 真实 agent-web-kit / agent-gateway / OpenCode 链路。
- 生产级完整审计。
- 复杂批量操作。
- 完整 Skill scripts/resources 文件资源管理。
- 生产级性能优化。
```

P0a 中的 `handledBy`、`handledAt` 等字段可由 mock 当前用户和本地时间模拟，只用于演示人在环闭环，不代表真实权限与审计体系已经完成。


### 3.2 P0a 原型验收参照

P0a 开发应以 `docs/prototypes/Agent-Team-V2-注册中心-v0.6.32.html` 作为闭环演示参照，而不是只参考静态页面展示。

原型验收重点：

| 原型动作 | 对应 DoD | 期望结果 |
|---|---|---|
| 保存并发布 Skill | DoD-04 / DoD-05 | Skill 版本变化，影响范围刷新，模板 / 员工派生技能可见。 |
| 关联 / 移除岗位技能 | DoD-05 | TemplateSkillMapping 更新，岗位技能数和员工派生技能刷新。 |
| 基于岗位创建数字员工 | DoD-06 / DoD-07 / DoD-07a | Worker 新增，保留 `agentTemplateId`，可加入团队并匹配项目；mock runtimeBinding 展示工作目录与技能同步状态。 |
| 普通员工通过 Leader 联系 | DoD-11 | 小云 target 为所属团队 Leader，HostContext 包含 workerId，不创建 Worker 直接会话。 |
| 发布项目文档 | DoD-08 / DoD-10 | Doc version 更新，Activity 新增，可生成模拟 Decision。 |
| 处理决策 | DoD-09 / DoD-10 | Decision 状态更新，待决策数量和 Activity 同步。 |
| 查看 P0a 演示主线面板 | 全部 DoD | 能按步骤确认最小闭环是否完成。 |

## 4. P0a 需求目标到开发任务追踪矩阵

| 需求目标 | 系统设计承接 | P0a 关键任务 | 验收方式 |
|---|---|---|---|
| 团队与员工管理 | `AgentTemplate`、`Worker`、`Team`、`ProjectAssignment` | `TEMPLATE-*`、`WORKER-*`、`TEAMPROJ-001` | 基于模板创建员工，员工加入团队并在总览 / 团队 / 项目中同步可见。 |
| 数字员工运行时绑定 | `RuntimeHost`、`WorkerRuntimeBinding`、`workspaceDir`、技能初始化快照 | `RUNTIME-*`、`WORKER-*` | 每个 Worker 能展示所属运行节点、工作目录、skillDir、初始化 SkillVersion 和 syncStatus；P0a 不真实写文件。 |
| 数字员工模板与技能继承 | `Skill`、`SkillVersion`、`TemplateSkillMapping`、`Worker.skills[]` 派生视图 | `SKILL-*`、`TEMPLATE-*`、`WORKER-003` | 技能发布后，模板和员工派生技能正确更新；草稿不影响运行中实例。 |
| 文档驱动执行 | `Doc`、`DocVersion`、发布状态、文档历史 | `DOC-001` | 保存草稿和发布是两个动作；发布后生成历史记录和 Activity。 |
| 决策闭环 | `Decision`、`Activity`、状态流转 | `DECISION-*`、`ACTIVITY-001` | pending 决策可处理为 resolved / blocked；处理后待办数量和动态回写正确。 |
| 小云连接能力 | `chat-integration`、`MockChatAdapter`、`HostContext` | `CHAT-009` | 从普通员工卡发起沟通时打开所属团队 Leader target，HostContext 包含 workerId，不创建 worker 直接会话。 |
| 运行态观察 | 总览 / 团队 / 项目读侧投影 | `TEAMPROJ-001`、`ACTIVITY-001`、`QA-001` | 关键业务动作能投影到运行态页面，页面截图通过走查。 |
| 配置态管理 | 岗位 / 员工 / 技能 / 设置 | `NAV-*`、`SKILL-*`、`TEMPLATE-*`、`WORKER-*` | 配置区导航清楚，设置页保持只读，配置能力不混入运行态页面。 |

## 5. P0a 开发顺序

P0a 不按“页面完整度”优先，而按“最小闭环主线”优先。

```text
第一层：不可跳过的基础骨架任务
→ 第二层：最小闭环任务
→ 第三层：页面补全与体验增强任务
```

约束：

```text
第一层未完成前，不进入页面功能开发。
第二层未跑通前，不做视觉增强、复杂筛选、批量操作或真实集成。
```

### 5.1 第一层：不可跳过的基础骨架任务

- 建立 pnpm workspace monorepo。
- 建立 `apps/web`。
- 建立 `packages/contracts`、`packages/domain`、`packages/mock-seed`、`packages/runtime-registry`、`packages/state`、`packages/data-client`、`packages/ui`、`packages/chat-integration/mock`。
- 建立 mock seed 和 mock data-client。
- 建立运行态 / 配置态导航与路由框架。

### 5.2 第二层：最小闭环任务

- Skill 工作台、草稿、发布生效。
- AgentTemplate 列表、模板技能匹配、创建 Worker 入口。
- Worker 列表、员工详情、派生技能展示、运行时绑定展示。
- Team / Project 状态投影。
- Doc 草稿 / 发布模拟。
- Decision 列表 / 处理 / 状态流转。
- Activity 回写。
- CHAT-009 普通员工通过 Leader 联系。
- E2E-P0A-001 最小闭环故事验收。

### 5.3 第三层：页面补全与体验增强任务

- 员工搜索筛选优化。
- 岗位技能跳转体验。
- 技能编辑体验优化。
- 总览 / 团队 / 项目视觉补全。
- 页面截图评审。

## 6. 推荐工程结构

开发计划与系统设计保持一致，采用 pnpm workspace monorepo。P0a 可以先创建前端和共享包，P0b 再补 API 与数据库。

```text
agent-team/
├── apps/
│   ├── web                         # P0a：前端应用
│   └── api                         # P0b：Fastify API
│
├── packages/
│   ├── contracts                   # 共享类型、DTO、枚举、API 契约
│   ├── domain                      # 领域模型、状态流转、派生规则
│   ├── mock-seed                   # P0a mock 数据
│   ├── runtime-registry            # RuntimeHost / WorkerRuntimeBinding 登记；P0a 可先由 mock 表达
│   ├── data-client                 # mock client / HTTP client 统一门面
│   ├── state                       # 前端状态管理和 selectors
│   ├── ui                          # 业务 UI 组件
│   └── chat-integration            # agent-web-kit 适配层
│
├── db/                             # P0b 引入
│   ├── schema.prisma
│   ├── migrations/
│   └── seed/
└── docs/
```

P0a 最小创建范围：

```text
apps/web
packages/contracts
packages/domain
packages/mock-seed
packages/runtime-registry
packages/state
packages/ui
packages/data-client
packages/chat-integration/mock
```

P0b 再创建：

```text
apps/api
db/schema.prisma
db/migrations
packages/data-client/http
```

## 7. 数据模型开发顺序

1. `AgentTemplate`：数字员工模板 / 岗位模板。
2. `Skill`：独立 Skill Markdown 能力资产。
3. `TemplateSkillMapping`：岗位模板与 Skill 的唯一匹配主数据。
4. `Worker`：数字员工实例，引用 `agentTemplateId`。
5. `RuntimeHost` / `WorkerRuntimeBinding`：数字员工到 OpenCode 工作目录的运行时绑定。
6. `ProjectAssignment`：实例与团队 / 项目的匹配关系。
6. `DocVersion`：草稿、发布版本和历史。
7. `Decision` / `Activity`：决策闭环和事件追踪。

## 8. 联调计划

### 8.1 Mock 阶段

- 使用 mock seed 初始化 teams、workers、agentTemplates、skills、mappings、docs、decisions、activities。
- 所有 UI 状态先完成本地联动。
- 完成截图评审：总览、员工、岗位、技能、团队详情、文档库、待决策。

### 8.2 API 阶段

- 先接只读 API：Team / Worker / AgentTemplate / Skill / Doc / Decision。
- 再接写 API：创建 Worker、保存 Skill 草稿、发布 Skill、发布文档、处理 Decision。
- 再接持久化：SkillVersion、DocVersion、Activity。
- 小云真实 agent-web-kit 文本对话作为可选项；完整 question / decision 回写放入 P1。

## 9. UI 评审与截图要求

每次迭代至少输出以下截图用于设计评审：

- 总览页：验证运行态入口与协作全景。
- 员工页：验证配置区中数字员工实例管理心智。
- 岗位页：验证“数字员工模板（岗位）”命名和创建链路。
- 技能页：验证 Skill 配置页是否接近 SKILL.md 工作台形态。
- 团队详情：验证运行态观察和配置区职责没有混淆。

截图评审要同时检查功能完整性和视觉层级，包括菜单分组、标题文案、主要按钮、影响提示、空状态和占位能力标记。
