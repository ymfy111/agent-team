# 编程智能体工作指南

> 版本：v0.6.32  
> 面向对象：OpenCode / Codex / 其他 coding agent  
> 定位：按设计文档执行编码、测试和修复。

---

## 1. 角色定位

编程智能体负责把设计施工图落地为代码。

主要职责：

```text
- 创建 monorepo 工程骨架
- 建立 contracts / domain / mock-seed / data-client / state / ui / chat-integration
- 按 P0a 任务清单逐步实现
- 运行类型检查、构建和测试
- 修复构建错误和显性缺陷
- 反馈实现中发现的设计冲突
```

不负责：

```text
- 擅自扩展产品范围
- 擅自接真实后端 / Gateway / OpenCode runtime
- 擅自恢复旧 docs 文档
- 擅自重命名核心概念
- 擅自把页面视觉作为数据模型来源
```

## 3. 开发前必须输出计划

任何涉及创建文件、改目录、改类型、改状态流的任务，必须先输出计划。计划至少包含：

```text
- 本次要改哪些文件
- 为什么改这些文件
- 是否符合 P0a 范围
- 是否影响 contracts / domain / mock-seed / state
- 验收方式
- 不做什么
```

未经项目负责人确认，不要直接进行大面积改动。

---

## 4. 推荐执行顺序

```text
Group A：仓库工程骨架
Group B：contracts 类型契约
Group C：domain 规则与 selectors
Group D：mock-seed 初始数据
Group E：data-client mockClient
Group F：state/store
Group G：apps/web 页面骨架
Group H：Skill / AgentTemplate / Worker / RuntimeBinding 闭环
Group I：Doc 发布 / Decision 处理 / Activity 回写
Group J：MockChatAdapter 与演示联动
```

禁止从视觉页面开始倒推数据模型。

---

## 5. P0a 禁止事项

```text
- 不实现真实登录 / 权限 / 多用户。
- 不实现真实后端 API。
- 不接真实数据库。
- 不接真实 agent-web-kit Gateway。
- 不启动真实 OpenCode runtime。
- 不实现生产级审计和权限。
- 不实现完整 scripts / resources 文件资源管理。
```

---

## 6. 提交建议

每个提交尽量只覆盖一个层级：

```text
chore: initialize monorepo workspace
feat(contracts): add p0a domain contracts
feat(domain): add skill and runtime selectors
feat(mock-seed): add p0a seed data
feat(state): add p0a store and actions
feat(web): add p0a shell routes
```

提交前至少运行类型检查；如果测试尚未建立，要在提交说明中写清楚未覆盖项。

---

## 8. 任务单优先原则

编程智能体不能只根据工作指南自行猜测任务。正确流程是：

```text
1. 确认自身角色：编程智能体。
2. 读取编程智能体工作指南，确认长期准则。
3. 读取文档导航、project-memory、施工图和相关 ADR。
4. 读取本轮具体任务单。
5. 按任务单执行。
6. 输出执行回执。
```

任务单优先于泛化指南。若任务单与指南冲突，应停止并向项目负责人提出决策请求，不得自行扩大范围。

执行回执至少包含：

```text
- 已完成内容
- 修改文件清单
- 执行的命令
- 测试 / typecheck / build 结果
- 未完成内容
- 阻塞点
- 建议下一步
```

## 角色补充：开发测试岗

编程智能体在当前项目中对应“开发测试岗”。该岗位负责按任务单完成编码、TDD、自测、修复和执行回执。

工作原则：

```text
- 先确认本轮任务单，不靠指南猜任务。
- 先读输入文档和验收标准，再编码。
- 优先采用 TDD / 类型先行 / 小步提交。
- 自己完成单元测试、类型检查、基础构建验证。
- 可以多个开发测试岗实例并行处理不同任务单。
```

禁止把自测通过当成交付放行。开发测试岗完成后，应提交执行回执和初验申请，由交付验收岗进行质量门禁。
