# 智能体协作分工

> 版本：v0.6.33  
> 日期：2026-05-17  
> 状态：当前过渡期协作规则  
> 说明：这里的“智能体”指参与本项目建设的外部 AI 工作角色，不是智能软件工厂系统内部的业务对象。

---

## 1. 为什么需要这份分工

当前智能软件工厂系统还没有实现，尚不能自动完成：

```text
- 给智能体分配岗位
- 给智能体分配项目
- 给智能体下发上下文
- 协调多个智能体按职责协作
- 追踪文档、决策、任务、代码产出之间的关系
```

因此，当前阶段仍由项目负责人手动调度外部智能体协作。项目仓库中的文档就是共同工作区。

---

## 2. 当前参与角色

| 角色 | 当前代表 | 主要职责 | 不负责 |
|---|---|---|---|
| 项目负责人 | 用户 | 决策、确认方向、在不同智能体之间搬运结果、提交 Git。 | 不负责逐行写全部设计或代码。 |
| 设计智能体 | ChatGPT 设计会话 | 需求梳理、系统设计、施工图、评审、文档治理、交接。 | 不直接承担最终编码和本地测试。 |
| 编程智能体 | OpenCode / Codex | 按设计文档创建文件、编码、跑测试、修复构建问题。 | 不扩展产品概念，不擅自改核心边界。 |
| 评审智能体 | 独立复审视角 / 新会话 | 按 MUST / SHOULD / COULD 复审设计、计划、代码变更。 | 不默认认可上一轮方案。 |

---

## 3. 当前人工调度流程

```text
1. 项目负责人提出目标或问题。
2. 设计智能体形成需求、设计、施工图、评审意见或文档包。
3. 项目负责人确认后同步到 GitHub。
4. 编程智能体读取 README.md、docs/文档导航.md、project-memory 和施工图。
5. 编程智能体按任务清单编码、测试、修复。
6. 需要时引入评审智能体复审设计或代码结果。
7. 项目负责人决定是否合并、提交、继续迭代。
```

---

## 4. 与智能软件工厂产品价值的关系

当前人工协作方式暴露了智能软件工厂要解决的问题：

```text
没有智能软件工厂：
  人要反复告诉每个智能体它是谁、负责什么、该读哪些文档、不能越界做什么。
  智能体之间没有统一的岗位、项目、任务、文档和决策上下文。
  协作靠人搬运上下文和同步结果。

有智能软件工厂：
  系统为每个数字员工明确岗位、技能、项目、团队和运行时绑定。
  智能体进入项目后知道自己的职责边界、上下文来源和协作对象。
  文档发布、决策处理、任务进展和活动回写成为统一链路。
  人从手动调度智能体转变为管理智能体团队和关键决策。
```

## 5. 术语边界

本目录 `docs/collab/` 描述的是当前项目建设过程中的外部智能体协作方式。

智能软件工厂业务模型中，应使用以下术语：

```text
岗位 = AgentTemplate / 数字员工模板
数字员工 = Worker / AgentInstance
技能 = Skill / SkillVersion
岗位技能匹配 = TemplateSkillMapping
运行时绑定 = WorkerRuntimeBinding
```

不要把 `docs/collab/` 中的“设计智能体 / 编程智能体 / 评审智能体”误认为系统内部已经实现的岗位实例。
