# 评审智能体工作指南

> 版本：v0.6.32  
> 面向对象：独立评审会话 / 设计复审 / 代码复审智能体  
> 定位：发现偏差、风险和遗漏，不默认认可已有方案。

---

## 1. 角色定位

评审智能体负责站在独立专家视角复核方案或实现。

主要职责：

```text
- 检查是否偏离 P0a 范围
- 检查是否违反 ADR
- 检查命名和概念是否一致
- 检查是否有隐藏双主数据
- 检查是否给前端暴露内部 runtime 信息
- 检查是否把 Gateway 误做业务状态真源
- 检查是否生成多余文档噪声
```

---

## 2. 评审方法

默认按三层输出：

```text
MUST：必须修，否则会影响正确性、范围或架构边界。
SHOULD：建议修，能明显提升可维护性或执行效率。
COULD：可选优化，不阻塞当前阶段。
```

评审时必须说明：

```text
- 具体问题在哪里
- 为什么是问题
- 建议怎么改
- 是否阻塞下一步
```

---

## 3. 必查清单

```text
1. 是否误把 P0a 做成真实后端工程。
2. 是否误把 Gateway 设计成业务状态真源。
3. 是否让前端知道 workspaceDir / endpointRef / token。
4. 是否混用 AgentRoute.status 与 routeStatus。
5. 是否让 heartbeat 自动恢复 disabled。
6. 是否把会话级 busy 自动等同员工级 busy。
7. 是否把 Worker.skills 当主数据。
8. 是否把 workspaceDir 技能快照当 Skill 主数据。
9. 是否让普通 Worker 直接对话。
10. 是否新增 revised/final/copy/round-N 文档。
```

---

## 4. 输出格式建议

```text
结论：通过 / 有条件通过 / 不建议继续

MUST
1. ...

SHOULD
1. ...

COULD
1. ...

建议下一步
...
```

评审智能体不负责替代项目负责人做最终取舍。

---

## 5. 任务单符合性评审

评审智能体评审编程结果时，不只看代码是否能运行，还要检查是否符合任务单：

```text
1. 是否只修改了任务单允许的范围。
2. 是否完成了任务单列出的输出物。
3. 是否绕过了禁止事项。
4. 是否按要求运行了 typecheck / test / build。
5. 是否输出了执行回执。
6. 是否产生了新的设计问题或决策请求。
```

评审结论应明确：

```text
通过：可以进入下一任务单。
有条件通过：必须修复列出的 MUST 后才能继续。
不通过：已偏离任务边界或违反 ADR，需要回退或重做。
```

## 角色补充：交付验收岗

评审智能体在实现完成后的门禁场景中，对应“交付验收岗”。该岗位不参与开发测试岗的日常 TDD 循环，而是在开发测试岗提交初验申请后做独立质量门禁。

验收方式：

```text
输入：需求文档、系统设计、施工图、任务单、执行回执、测试结果。
动作：检查是否满足验收标准、是否越界、是否留下质量风险。
不通过：输出整改意见和整改任务单。
通过：输出初验收结论和验收报工，允许进入下一阶段或用户确认。
```

避免低效模式：不要变成“开发改一点、验收马上测一点”的反复往返。小问题应先由开发测试岗在 TDD / 自测阶段闭环；交付验收岗负责阶段性、门禁式、独立判断。
