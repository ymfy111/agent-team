# 需求 / 设计文档编写与多角色评审方法

> 适用范围：产品需求、系统设计、技术方案、集成方案、开发计划、验收用例等项目文档  
> 适用对象：产品经理、架构师、开发人员、测试人员、AI 智能体  
> 目标：让文档从“一次性输出”变成“可评审、可质疑、可迭代、可落地”的协作产物。

---

## 1. 核心原则

### 1.1 先读现有资料，再写新内容

编写文档前，应先阅读已有资料：

```text
- 当前需求文档
- 当前设计文档
- 原型 / 截图 / 示例
- 既有开发计划
- 既有评审意见
- 相关约束和历史决策
```

不要只凭印象或单条指令直接重写。

### 1.2 先明确本轮目标

每次文档工作开始前，先判断本轮属于哪类任务：

```text
- 新写需求
- 补充需求
- 新写设计
- 优化设计
- 写技术方案
- 写集成方案
- 写开发计划
- 写验收用例
- 做文档评审
- 做一致性检查
```

一轮只解决少数明确目标，避免把文档越改越散。

### 1.3 小步迭代，不轻易推翻

已有文档结构稳定时，优先做增量补充：

```text
- 补章节
- 统一术语
- 修正冲突
- 增加验收条件
- 补数据模型
- 补边界说明
```

除非现有结构明显错误，否则不要轻易整体重写。

---

## 2. 标准工作流

```text
读取资料
→ 明确目标
→ 生成初稿
→ 多角色评审
→ 识别必须修改项
→ 修订文档
→ 输出最终版
→ 记录遗留问题
```

### 2.1 读取资料

重点看：

```text
- 业务目标是什么
- 用户是谁
- 当前范围是什么
- 哪些是已定决策
- 哪些是待决策问题
- 哪些内容彼此冲突
- 哪些地方还只是占位
```

### 2.2 生成初稿

初稿应做到：

```text
- 结构清楚
- 术语一致
- 范围明确
- 逻辑可追踪
- 不混淆需求与设计
- 不混淆当前阶段与未来阶段
```

### 2.3 多角色评审

初稿完成后，不要直接交付，应从多个角度质疑它。

推荐评审角色：

```text
1. 产品评审
2. 架构评审
3. 前端评审
4. 后端评审
5. 数据模型评审
6. 集成评审
7. 测试 / 验收评审
8. 文档质量评审
```

### 2.4 修订文档

把评审意见分为三类：

```text
必须修改：影响理解、开发、验收或架构边界
建议修改：提升清晰度、完整性或可维护性
暂不修改：合理但不属于当前阶段
```

先处理必须修改项，再处理建议修改项。

## 3. 多角色评审清单

## 3.1 产品评审

检查问题：

```text
- 用户是谁？
- 用户要完成什么任务？
- 需求是否描述了业务价值？
- 页面 / 功能是否符合用户心智？
- MVP 范围是否过大？
- 是否混入了暂不需要的功能？
- 术语是否容易误解？
```

常见质疑：

```text
- 这个功能为什么现在必须做？
- 用户是否能理解这个入口？
- 这是核心需求，还是未来增强？
- 是否存在同一个概念多个叫法？
```

---

## 3.2 架构评审

检查问题：

```text
- 模块边界是否清楚？
- 谁拥有主数据？
- 谁只是展示或派生？
- 是否出现循环依赖？
- 是否把外部系统职责写进了当前系统？
- 是否为后续扩展保留了合理边界？
```

常见质疑：

```text
- 这个模块是不是承担了太多职责？
- 这个能力应该在当前系统实现，还是通过外部组件集成？
- 这里有没有重复造轮子？
- 这个设计是否方便后续替换实现？
```

---

## 3.3 前端评审

检查问题：

```text
- 页面结构是否清楚？
- 路由和导航是否合理？
- 页面状态如何管理？
- 哪些是全局状态，哪些是局部状态？
- 组件是否可以拆分？
- 原型里的占位能力是否被误认为已完成？
```

常见质疑：

```text
- 这个页面是否信息过密？
- 这个交互是否能在当前阶段实现？
- 是否需要先用 mock 数据验证？
- 用户完成关键链路需要点几步？
```

---

## 3.4 后端评审

检查问题：

```text
- 是否需要后端支持？
- API 边界是否清楚？
- 数据是否需要持久化？
- 是否需要审计记录？
- 是否需要权限控制？
- 是否需要异步任务或事件机制？
```

常见质疑：

```text
- 当前阶段是否真的需要真实后端？
- 这个接口是查询、命令，还是事件？
- 失败后如何恢复？
- 数据一致性由谁保证？
```

---

## 3.5 数据模型评审

检查问题：

```text
- 核心对象是否清楚？
- 对象之间的关系是否明确？
- 是否存在重复主数据？
- 哪些字段是派生字段？
- 状态枚举是否清晰？
- 草稿、发布、历史是否分离？
```

常见质疑：

```text
- 这个字段应该存储，还是实时计算？
- 两个对象是否表达了同一件事？
- 状态流转是否完整？
- 删除、停用、归档的语义是否区分？
```

---

## 3.6 集成评审

检查问题：

```text
- 当前系统与外部系统的边界是什么？
- 谁主动调用谁？
- 数据通过什么协议传递？
- 失败时如何降级？
- 是否有 mock / fallback 方案？
- 外部系统是否反向依赖了当前系统业务模型？
```

常见质疑：

```text
- 集成是否足够非侵入？
- 是否把外部组件嵌得太深？
- 是否存在双重状态管理？
- 是否能在外部系统不可用时保持主流程可用？
```

---

## 3.7 测试 / 验收评审

检查问题：

```text
- 文档是否能转成测试用例？
- 每个核心功能是否有验收条件？
- 是否覆盖端到端用户故事？
- 是否覆盖异常状态？
- 是否明确哪些能力只是占位？
```

常见质疑：

```text
- 如何证明这个功能完成了？
- 用户从入口到结果的完整路径是什么？
- 空数据、失败、超时、权限不足时怎么办？
- 这个需求是否可测试？
```

---

## 3.8 文档质量评审

检查问题：

```text
- 标题层级是否清楚？
- 是否有重复内容？
- 是否有前后矛盾？
- 术语是否统一？
- 表格和列表是否可读？
- 是否区分事实、设计建议和待决策问题？
```

常见质疑：

```text
- 这段话是在描述需求，还是设计实现？
- 这个结论是否有依据？
- 这个章节是否放错位置？
- 是否需要补一张表或流程图？
```

## 4. 文档类型建议

### 4.1 需求文档

重点回答：

```text
- 谁使用？
- 解决什么问题？
- 有哪些功能？
- 用户如何操作？
- 什么算完成？
```

推荐结构：

```text
1. 背景与目标
2. 用户与场景
3. 范围说明
4. 核心概念
5. 功能清单
6. 页面 / 交互需求
7. 业务规则
8. 验收标准
9. 非目标 / 暂不做
10. 待决策问题
```

---

### 4.2 设计文档

重点回答：

```text
- 系统如何实现？
- 模块如何拆分？
- 数据如何流转？
- 状态如何变化？
- 外部系统如何集成？
```

推荐结构：

```text
1. 设计目标
2. 设计原则
3. 总体架构
4. 模块边界
5. 数据模型
6. 状态流转
7. 关键业务流程
8. API / 集成方式
9. 持久化设计
10. 错误与降级
11. 测试策略
12. 待决策问题
```

---

### 4.3 技术方案 / 集成方案

重点回答：

```text
- 为什么需要这个方案？
- 集成边界是什么？
- 接入步骤是什么？
- 双方职责是什么？
- 有哪些风险和注意事项？
```

推荐结构：

```text
1. 背景
2. 目标
3. 非目标
4. 总体方案
5. 职责边界
6. 接入步骤
7. 数据 / 协议约定
8. 错误与降级
9. 分阶段计划
10. 风险与注意事项
```

---

### 4.4 开发计划

重点回答：

```text
- 先做什么？
- 后做什么？
- 依赖关系是什么？
- 每一步如何验收？
```

推荐结构：

```text
1. 开发目标
2. 阶段划分
3. 任务拆解
4. 依赖关系
5. 验收标准
6. 风险与缓解
7. 后续计划
```

## 5. 评审报告模板

```markdown
# 文档评审报告

## 1. 评审结论

- 是否建议合入：
- 是否存在阻塞问题：
- 本轮主要风险：

## 2. 分角色评审意见

### 2.1 产品评审

### 2.2 架构评审

### 2.3 前端评审

### 2.4 后端评审

### 2.5 数据模型评审

### 2.6 集成评审

### 2.7 测试 / 验收评审

### 2.8 文档质量评审

## 3. 必须修改项

| 编号 | 问题 | 影响 | 处理结果 |
|---|---|---|---|

## 4. 建议修改项

| 编号 | 建议 | 价值 | 是否采纳 |
|---|---|---|---|

## 5. 已采纳修改

## 6. 暂不采纳项

## 7. 遗留问题

## 8. 下一步建议
```

## 6. 通用提示词

可将以下提示词交给 AI 智能体使用。

```text
你正在协助编写和评审项目文档。

请遵守以下流程：

1. 先阅读当前已有文档、原型、上下文和约束。
2. 明确本轮目标，不要一次性解决过多问题。
3. 保持既有术语和结构，除非发现明确冲突。
4. 区分需求、设计、实现、验收和待决策问题。
5. 初稿完成后，必须从以下角色进行评审：
   - 产品评审
   - 架构评审
   - 前端评审
   - 后端评审
   - 数据模型评审
   - 集成评审
   - 测试 / 验收评审
   - 文档质量评审
6. 将评审意见分为：
   - 必须修改
   - 建议修改
   - 暂不修改
7. 根据评审意见修订文档。
8. 输出最终文档时，同时给出：
   - 修改摘要
   - 主要评审意见
   - 已采纳修改
   - 遗留问题
   - 下一步建议
```

## 7. 禁止事项

```text
- 未阅读现有资料就直接重写
- 把需求和实现混在一起
- 把未来能力写成当前已完成能力
- 把占位功能写成正式能力
- 多个名称表达同一个概念
- 多个对象维护同一份主数据
- 只写功能，不写验收标准
- 只写正常流程，不写异常状态
- 只给结论，不解释边界和依据
```

---

## 8. 简化版流程

如果时间有限，至少执行以下步骤：

```text
1. 读现有资料
2. 写初稿
3. 从产品、架构、测试三个角度自查
4. 修正冲突和缺口
5. 输出最终版
6. 记录遗留问题
```
