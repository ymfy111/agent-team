# 文档维护规则

> 版本：v0.6.33  
> 日期：2026-05-17  
> 状态：当前文档治理规则

---

## 1. 为什么需要规则

v0.6.32 前后产生了多个版本稿、修订稿、评审稿、原型稿和参考稿。它们对探索有价值，但放在当前 `docs/` 正本目录中会干扰 OpenCode 读取。

v0.6.33 起，`docs/` 目录只保留“当前可执行正本”。

---

## 2. 命名规则

当前正本命名：

```text
YYYY-MM-DD-主题-v版本.md
```

禁止后缀：

```text
-revised
-final
-final2
-copy
-new
-修改版
-临时
```

需要大修时，直接生成新版本号：

```text
v0.6.33 → v0.6.34
```

---

## 3. 目录职责

```text
specs/      正式需求、系统设计、集成方案、Mock 设计、施工图
plans/      阶段路线图
tasks/      OpenCode 可执行任务清单
decisions/  ADR 架构决策
guides/     协作规范、计划模板、文档规则
references/ 外部项目分析，不作为正式设计真源
changes/    版本变更说明
reports/    合并后的评审结论，不保存每轮过程稿
prototypes/ 当前只放原型说明，不放多代 HTML 历史包
```

---

## 4. 历史材料处理

旧版本材料进入 Git 历史，不进入当前正本目录。

如果确实需要保留历史材料：

```text
- 建议通过 Git tag / branch 管理。
- 或放到仓库根级 archive/，不要放在 docs 当前正本目录。
- 不要让 OpenCode 默认读取 archive。
```

---

## 5. 新增文档门槛

只有满足以下条件才新增文档：

```text
- 它是一个新的正式主题，不适合合并到现有正本。
- 它会长期被 OpenCode 或项目负责人读取。
- 它有明确归属目录。
- 它不会与已有文档形成重复真源。
```

否则应修改现有正本。

---

## 6. 文档真源优先级

```text
project-memory.md
→ specs 当前正本
→ decisions ADR
→ tasks 当前执行清单
→ guides 协作规则
→ references 外部参考
→ reports 评审结论
```

外部参考和评审报告不能覆盖 specs/decisions 的正式结论。
