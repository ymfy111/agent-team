# 智能软件工厂文档维护规则

> 版本：v0.6.33  
> 日期：2026-05-17  
> 状态：当前规则

---

## 1. 目标

控制文档数量，避免每轮评审、每次小改都生成新副本，导致 OpenCode / 新会话不知道应该读哪一份。

v0.6.33 之后，文档维护目标是：

```text
入口少
正本清楚
历史靠 Git
重大决策写 ADR
执行内容写任务清单
```

---

## 2. 入口规则

保留：

```text
README.md
AGENTS.md
docs/文档导航.md
docs/project-memory.md
```

不再保留：

```text
docs/README.md
docs/**/README.md
```

原因：README 和文档导航重复会形成双入口。`docs/文档导航.md` 是唯一详细索引。

---

## 3. 正本规则

同一主题只保留一个当前正本：

```text
产品需求规格-v0.6.33.md
系统设计方案-v0.6.33.md
Mock数据设计-v0.6.33.md
agent-web-kit集成方案-v0.6.33.md
P0a工程施工图-v0.6.33.md
```

不要新增：

```text
*-revised.md
*-final.md
*-v2.md
*-copy.md
*-round-1.md
*-reviewed.md
```

如果只是小改，直接改当前正本，用 Git commit 追踪。

---

## 4. 历史规则

历史版本、过程稿、多轮评审稿，不再复制保留在当前 `docs/` 正本目录。

需要追溯时使用：

```bash
git log -- docs/
git show <commit>:docs/...
```

---

## 5. ADR 规则

遇到以下情况必须写 ADR：

```text
- 改变系统边界
- 改变状态真源
- 改变核心数据模型
- 改变 P0a / P0b / P1 范围
- 改变前后端或 Gateway 安全边界
```

ADR 一旦 Accepted，其他设计文档不得与其冲突。

---

## 6. reports 规则

`docs/reports/` 只保留当前阶段的收敛报告，不保留多轮过程稿。

例如：

```text
docs/reports/2026-05-17-v0.6.33文档瘦身与合并报告.md
```

如果继续评审，更新该报告或生成下一个里程碑报告，不要产生一堆 review-round-N 文件。

---

## 7. prototypes 规则

v0.6.33 当前阶段不把多代 HTML 原型作为 docs 正本保留。

如果后续需要视觉参照，可以在单独任务中恢复一个当前参考原型，并命名为：

```text
docs/prototypes/Agent-Team-V2-注册中心-v0.6.33-reference.html
```

旧原型通过 Git 历史追溯，不应批量放回当前 docs。

---

## 8. 打包规则

生成给用户覆盖的包时，优先包含：

```text
README.md
AGENTS.md
MIGRATION-v0.6.33-docs正本化.md
docs/
```

如果只修改 docs，也必须说明根 README 是否需要同步。


## 智能体协作文档位置

设计智能体、编程智能体、评审智能体的分工与工作指南统一放在 `docs/collab/`，不要放在根目录 `AGENTS.md`，也不要混入业务模型中的“岗位 / Worker / AgentTemplate”说明。

本次 v0.6.33 是过期文档清理，可全量替换 `docs/`；后续日常工作只更新当前任务实际变化的文档。
