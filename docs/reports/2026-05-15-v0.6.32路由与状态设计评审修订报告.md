# v0.6.32 路由与状态设计评审修订报告

> 日期：2026-05-15  
> 状态：评审修订完成  
> 范围：数字员工启用 / 停用、AgentRoute、AgentPresence、AgentRouteProvider、RuntimeHost / Agent 心跳，以及 agent-web-kit 路由补充设计对齐。

## 1. 修订结论

本轮确认以下设计原则：

```text
正式智能软件工厂中：
Worker / RuntimeHost / AgentRoute / AgentPresence 的业务真源在 agent-team。

agent-web-kit Gateway：
只负责 agentId 解析、会话创建、命令转发、SSE 订阅和轻量状态摘要。

轻量 / Mock / Demo 场景：
Gateway 可以自行维护临时 route registry 和 heartbeat 状态。
```

## 2. 已处理问题

| 编号 | 问题 | 处理结果 |
|---|---|---|
| MUST-001 | agent-web-kit 文档需保持通用性，agent-team 只能作为示例 | 已在 agent-web-kit v4 中将正式业务系统抽象为宿主业务系统 |
| MUST-002 | route 更新后已有 conversation 绑定策略不明确 | 已补 routeRevision 与 ConversationRef 绑定策略 |
| MUST-003 | internal CRUD / heartbeat 接口安全需要硬约束 | 已补默认关闭、service token / mTLS / allowlist、生产显式启用 |
| SHOULD-001 | RuntimeHost 心跳与 Agent 心跳关系不明确 | 已补 Host offline 对 agent route / presence 的推导规则 |
| SHOULD-002 | Provider 缺少批量 resolve | 已补 `POST /internal/agent-routes/resolve-batch` |
| SHOULD-003 | DELETE route 对 active conversation 行为不明确 | 已补 `force=false` 与 `409 ROUTE_IN_USE` |
| SHOULD-004 | Static 模式缺少 reload | 已补 internal reload 建议 |
| SHOULD-005 | resolveAgent 状态类型未标准化 | 已补 PublicAgentStatus |
